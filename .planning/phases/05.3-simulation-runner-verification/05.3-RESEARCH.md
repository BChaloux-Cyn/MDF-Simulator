## Summary

Phase 5.3 is an integration phase: the compiler (5.2) and the runtime framework (5.1) are both complete and tested. The primary work is writing the glue that connects them at runtime, plus defining and validating the scenario input format, and verifying correctness end-to-end against the elevator model.

The most significant technical finding is a **ctx API gap**: generated code calls `ctx.generate("Event", target=self_dict["__instance_key__"], args={})` (and similarly for `ctx.create`, `ctx.delete`, `ctx.relate`, `ctx.cancel`) using an `__instance_key__` tuple stored in the instance dict, but the current `engine/ctx.py` API takes separate `sender_class/sender_id/target_class/target_id` arguments and does not populate `__instance_key__` in instance dicts at all. Phase 5.3 must close this gap before the elevator scenario can run.

The bundle loader is conceptually straightforward: extract zip to temp dir, read `bundle.json` for version check, load `manifest.json` and deserialize (reversing `state::event` string keys back to `(state, event)` tuples), import each `generated/<Class>.py` module via `importlib.util.spec_from_file_location`, extract the `TRANSITION_TABLE` dict (which already has live callables), and use those callables to populate the `ClassManifest` TypedDicts.

**Primary recommendation:** Before any other work, extend `engine/ctx.py` to accept the generated-code-facing API (`target=instance_key`, `ctx.create`, `ctx.delete`, `ctx.relate`, `ctx.cancel`, `ctx.select_any_related`, etc.) and populate `__instance_key__` and `__class_name__` in every instance dict at creation time. All other Phase 5.3 work depends on this being correct.

### Pattern 1: Bundle Loader

**What:** Extracts `.mdfbundle` zip to a temp dir, verifies engine version, loads `manifest.json`, imports generated `.py` files, and rebinds callables into `ClassManifest` TypedDicts.

**Key steps:**
```python
# Source: compiler/packager.py (bundle layout), engine/manifest.py (TypedDicts)

import importlib.util, json, tempfile, zipfile
from pathlib import Path

def load_bundle(bundle_path: Path) -> tuple[dict, Path]:
    """Returns (DomainManifest-with-callables, temp_dir_path)."""
    tmpdir = tempfile.mkdtemp(prefix="mdfbundle_")
    with zipfile.ZipFile(bundle_path) as zf:
        zf.extractall(tmpdir)
    
    bundle_meta = json.loads((Path(tmpdir) / "bundle.json").read_text())
    _check_engine_version(bundle_meta["engine_version"])
    
    manifest = json.loads((Path(tmpdir) / "manifest.json").read_text())
    # manifest.json has "state::event" string keys — must reverse to (state, event) tuples
    _restore_transition_table_keys(manifest)
    
    # Import generated/<Class>.py and rebind callables
    generated_pkg = Path(tmpdir) / "generated"
    for cls_name, cls_manifest in manifest["class_defs"].items():
        module = _import_module_from_file(
            f"mdf_generated_{cls_name}", 
            generated_pkg / f"{cls_name}.py"
        )
        tt = getattr(module, "TRANSITION_TABLE", {})
        for key, entry in tt.items():
            manifest["class_defs"][cls_name]["transition_table"][key]["action_fn"] = entry.get("action_fn")
            manifest["class_defs"][cls_name]["transition_table"][key]["guard_fn"] = entry.get("guard_fn")
    
    return manifest, Path(tmpdir)
```

**Manifest key reversal:** `packager.py` serializes `(state, event)` tuple keys as `"state::event"` strings (see `_strip_callables`). The loader must split on `"::"` to reconstruct tuples. [VERIFIED: compiler/packager.py lines 43-44]

### Pattern 3: ctx API Gap — Generated Code vs Runtime API

**This is the most critical finding in this research.**

The compiler (`transformer.py`) generates code that calls:
- `ctx.generate("Event", target=self_dict["__instance_key__"], args={...}, delay_ms=...)` [VERIFIED: transformer.py:352-353]
- `ctx.cancel("Event", sender=sender_inst, target=target_inst)` [VERIFIED: transformer.py:376]
- `ctx.create("Class", {params})` → returns instance dict [VERIFIED: transformer.py:421-422]
- `ctx.delete(var)` → deletes instance by reference [VERIFIED: transformer.py:429]
- `ctx.delete_where("Class", lambda inst: cond)` [VERIFIED: transformer.py:433]
- `ctx.relate(a, b, "R1")` → a and b are instance dicts [VERIFIED: transformer.py:507]
- `ctx.unrelate(a, b, "R1")` → a and b are instance dicts [VERIFIED: transformer.py:514]
- `ctx.select_any("Class", where=lambda inst: ...)` [VERIFIED: transformer.py:449]
- `ctx.select_many("Class", where=lambda inst: ...)` [VERIFIED: transformer.py:450]
- `ctx.select_any_related(source, ["R1", "R2"])` [VERIFIED: transformer.py:464]
- `ctx.select_many_related(source, ["R1", "R2"])` [VERIFIED: transformer.py:464]
- `ctx.bridge("Domain", "Op", {args})` [VERIFIED: transformer.py:401]
- `ctx.traverse(source, ["R1", "R2"])` — internal to select_related [VERIFIED: transformer.py:287]

The current `engine/ctx.py` has mismatched signatures:
- `ctx.generate` requires `sender_class, sender_id, target_class, target_id` as separate positional args — NOT `target=instance_key` [VERIFIED: ctx.py:52-73]
- `ctx.relate` requires `rel_id, class_a, id_a, class_b, id_b` — NOT `(a_dict, b_dict, "R1")` [VERIFIED: ctx.py:128-136]
- `ctx.create`, `ctx.delete`, `ctx.delete_where`, `ctx.select_any_related`, `ctx.select_many_related`, `ctx.traverse` do not exist at all [VERIFIED: ctx.py]

**Root cause:** `__instance_key__` is a per-instance dict slot that was planned (transformer.py references it) but never populated by the registry. Instance dicts created by `InstanceRegistry.create_sync` do not include `__instance_key__` or `__class_name__`. [VERIFIED: registry.py:107-119]

**Resolution required:** Either extend `ctx.py` with generated-code-facing overloads/new methods, or add a thin adapter layer. The cleanest approach given the existing design is:
1. Add `__instance_key__` and `__class_name__` to every instance dict at registry creation time.
2. Extend `ctx.generate` to accept `target=instance_dict_or_key` (instance-dict path).
3. Add `ctx.create(class_name, params)` → returns instance dict with `__instance_key__` and `__class_name__` set.
4. Add `ctx.delete(instance_dict)` → uses `inst["__class_name__"]` + `inst["__instance_key__"]`.
5. Add `ctx.relate(a_dict, b_dict, rel_id)` → extracts class/id from instance dicts.
6. Add `ctx.select_any_related`, `ctx.select_many_related`, `ctx.traverse` for navigation.

This extension should go in `engine/ctx.py` alongside the existing methods (same object, different signatures for different callers).

def simulate_domain(domain: str, scenario: str) -> dict:

### Pattern 6: Trigger Evaluation

**What:** After each micro-step, evaluate all armed triggers. [VERIFIED: CONTEXT.md D-18]

**Implementation location:** The trigger evaluator runs in the simulation loop between micro-steps, not inside the engine itself (engine is schema-free per D-37). It lives in the bundle loader / MCP tool layer.

**Approach:**
```python
for step in ctx.execute():
    yield step
    # evaluate triggers after each micro-step
    for trigger in armed_triggers:
        if _trigger_matches(trigger, ctx.registry):
            if not trigger.repeat:
                armed_triggers.remove(trigger)
            _fire_trigger(trigger, ctx)
```

This means `simulate_domain` does not call `run_simulation` directly — it drives the `ctx.execute()` generator manually, interleaving trigger evaluation.

### Pattern 7: load_scenario Extension

**What:** Current `ctx.load_scenario()` does not handle aliases, relationships, time-based events, or triggers. Phase 5.3 extends or replaces this method. [VERIFIED: ctx.py:186-211]

**The current format** (`{"instances": [...], "events": [...]}`) uses raw class/identifier dicts, not aliases. The new scenario format uses aliases that must be resolved to instance dicts after creation.

**Resolution:** Either extend `ctx.load_scenario` with alias-aware logic, or do alias resolution in the tool layer before calling `ctx.load_scenario`. Given the complexity of triggers (which require post-step evaluation hooks), alias resolution in the tool layer is cleaner.

- **Calling `yaml.load` (unsafe):** Must use `yaml.safe_load` per existing T-5.1-01 security rule. [VERIFIED: bridge.py]
- **Importing engine in compiler:** D-11 prohibits `compiler/*` from importing `engine/*`. Phase 5.3 touches both, but in the bundle loader (engine side), not in compiler. [VERIFIED: compiler/__init__.py]
- **Rebuilding scenario aliases after simulation ends:** Aliases are only needed during setup. Don't keep them in the hot path.
- **Using exec() to load generated modules:** D-01 and D-03 explicitly reject exec. Use `importlib.util.spec_from_file_location`. [VERIFIED: CONTEXT.md D-01, compiler/codegen.py]
- **Writing trace files inside the temp dir:** Temp dir is cleaned up. Trace must go to `.design/traces/` before cleanup. [VERIFIED: CONTEXT.md D-07]

## Common Pitfalls

### Pitfall 1: Transition Table Key Format Mismatch
**What goes wrong:** `manifest.json` stores transition table keys as `"state::event"` strings. The engine TypedDicts and `ThreeQueueScheduler` use `(state, event)` tuple keys. Loading manifest.json without reversing this produces a manifest where no transitions can be looked up.
**Why it happens:** `packager.py:_strip_callables` deliberately converts tuple keys to strings for JSON serialization. [VERIFIED: packager.py:42-46]
**How to avoid:** In the loader, after `json.loads`, iterate `class_def["transition_table"]` and split each key on `"::"` to produce `(state, event)` tuples.
**Warning signs:** Every event produces `cant_happen` ErrorMicroStep even for valid transitions.

### Pitfall 2: `__instance_key__` Not Populated in Instance Dicts
**What goes wrong:** Generated action code accesses `self_dict["__instance_key__"]` and `self_dict["__class_name__"]` to call `ctx.generate`. These are not set by `InstanceRegistry._build_instance`. A `KeyError` at runtime kills the simulation.
**Why it happens:** Registry was built before the transformer's codegen convention was finalized. [VERIFIED: registry.py:50-81, transformer.py:322-324]
**How to avoid:** In `InstanceRegistry._build_instance` (or `create_sync`/`create_async`), set `instance["__instance_key__"] = make_instance_key(identifier)` and `instance["__class_name__"] = class_name`.

### Pitfall 3: ctx.generate Signature Mismatch
**What goes wrong:** Generated code calls `ctx.generate("Event", target=instance_dict["__instance_key__"], args={})`. Current `ctx.generate` requires `sender_class, sender_id, target_class, target_id` positional args. Calling with `target=` keyword raises `TypeError`.
**Why it happens:** The ctx API was designed for the scenario-loading use case (explicit class/id), not the generated-code use case (instance-dict reference).
**How to avoid:** Add a `ctx.generate` overload (or unified method) that accepts `target=instance_key_frozenset` and resolves class name from `__class_name__` stored in the instance dict.

### Pitfall 4: Generated Code Enum References Without Import
**What goes wrong:** Generated `Elevator.py` references bare names like `Up`, `Down`, `Direction` which are MDF enum members. These are rendered as Python names in the generated code but the enum class is not automatically imported in every generated file.
**Why it happens:** The transformer emits enum member names directly from the YAML model. [VERIFIED: Elevator.py lines 26-28 showing `Up`, `Down` without qualification]
**How to avoid:** Verify the codegen imports the relevant enum classes into generated modules when enum attributes appear in action bodies. This may already be handled — verify in `compiler/codegen.py` generate_class_module's type_registry rendering.

### Pitfall 5: Temp Dir Cleanup on Windows
**What goes wrong:** On Windows, `shutil.rmtree` fails if a module loaded from the temp dir is still referenced (Python holds a file handle via the module's `__spec__.origin`). This causes cleanup to raise `PermissionError`.
**Why it happens:** Windows does not allow deletion of open files.
**How to avoid:** After importing all modules and extracting callables, explicitly delete module references from `sys.modules` before calling `shutil.rmtree`. Or: use `errors='ignore'` on `shutil.rmtree` (acceptable — temp dir cleaned by OS on reboot regardless).

### Pitfall 6: `load_scenario()` Extension Breaking Existing Tests
**What goes wrong:** Existing `test_engine.py` tests use the old `load_scenario` dict format (`{"instances": [...], "events": [...]}` with raw class/identifier dicts). Changing the format breaks these tests.
**Why it happens:** Old format used explicit `{"class": X, "identifier": {...}}` not the new alias-based format.
**How to avoid:** Keep the old `load_scenario(scenario: dict)` signature intact; add a separate `load_scenario_file(path: str)` (or similar) that parses the new YAML format and resolves aliases. Don't change `run_simulation`'s signature. [VERIFIED: ctx.py:186-211, test_engine.py]

### Pitfall 7: Pre-Flight Multiplicity Check Requires Domain Manifest
**What goes wrong:** The pre-flight multiplicity check (D-13) needs the `AssociationManifest` for each relationship to know required multiplicities. This info is in `manifest.json`, but the scenario YAML refers to relationship IDs (e.g., `R1`). If the bundle isn't loaded yet, the check can't run.
**How to avoid:** Run the pre-flight check after bundle loading but before `ctx.execute()`. Pass `manifest["associations"]` into the check function.

import importlib.util
import sys

def _import_module_from_file(module_name: str, file_path) -> object:
    spec = importlib.util.spec_from_file_location(module_name, file_path)
    module = importlib.util.module_from_spec(spec)
    sys.modules[module_name] = module
    spec.loader.exec_module(module)
    return module

### Transition Table Key Reversal
```python
# Source: compiler/packager.py:42-46 (documents the encoding)
def _restore_transition_table(tt_serialized: dict) -> dict:
    """Reverse "state::event" string keys back to (state, event) tuples."""
    restored = {}
    for key_str, entry in tt_serialized.items():
        if "::" in key_str:
            state, event = key_str.split("::", 1)
            # Handle None state (creation events stored as "None::__creation__")
            restored[(None if state == "None" else state, event)] = entry
        else:
            restored[key_str] = entry  # passthrough for non-transition keys
    return restored
```

import json
from datetime import datetime
from dataclasses import asdict

def _write_trace(domain: str, scenario_path: str, steps: list) -> str:
    stem = Path(scenario_path).stem.replace(".scenario", "")
    timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")

## Open Questions

1. **Creation entry action key format in manifest.json**
   - What we know: packager converts `(None, "__creation__")` tuple keys. The serializer: `f"{k[0]}::{k[1]}"` → would produce `"None::__creation__"`.
   - What's unclear: Is `initial_state` also stored as a transition? Or is it only in `initial_state` field?
   - Recommendation: Test with the elevator bundle — the manifest.json should clarify. Check if `Door` class (which has async creation) has a `None::__creation__` entry.

2. **ctx.generate sender for generated code**
   - What we know: Transformer generates `ctx.generate("Event", target=..., args=...)` with no sender. Scheduler requires sender_class + sender_id.
   - What's unclear: Is the sender always `self` (current instance) in generated action code?
   - Recommendation: Yes — in xUML semantics, events generated within an action are always sent by the current instance. `ctx.generate` in generated-code mode should default sender to `self_dict`.

3. **Trigger evaluation and delayed events**
   - What we know: Triggers are evaluated after each micro-step (D-18). Delayed events are in the delay queue and don't produce micro-steps until expiry.
   - What's unclear: Should triggers be evaluated after `event_delayed` micro-steps? Or only after `transition_fired`?
   - Recommendation: Evaluate after every micro-step type. The trigger condition checks instance state/attrs, which only change on `transition_fired` and `action_executed`. Evaluating after every step is correct and cheap.
