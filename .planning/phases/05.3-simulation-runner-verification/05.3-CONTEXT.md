# Phase 5.3: Simulation Runner + Verification — Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Build the bundle loader that wires compiled `.mdfbundle` files into the runtime framework, verify the elevator model end-to-end with a multi-floor competing-call scenario, and expose `simulate_domain()` / `simulate_class()` as MCP tool wrappers in `tools/simulation.py`.

Also delivered in this phase: the scenario file format (Pydantic schema, YAML convention, `.design/scenarios/` location), conditional triggers, and trace file output.

Exit gate: Elevator model compiles, loads, runs a multi-instance multi-floor scenario, and produces the expected key micro-step milestones.

</domain>

<decisions>
## Implementation Decisions

### Bundle Loader

- **D-01:** Load generated Python modules by **extracting the bundle to a temp directory** and using `importlib.util.spec_from_file_location`. Stack traces show real file paths — critical for Phase 5.4/5.5 debugger work. Temp dir is cleaned up after simulation completes.
- **D-02:** **Hard fail** if `bundle.json`'s `engine_version` does not match the installed engine version. Clear error message with both versions. No partial loading, no warn-and-proceed.
- **D-03:** After extraction, the loader imports each `generated/<Class>.py`, extracts action and guard functions by name, and rebinds them into the `ClassManifest` TypedDicts before passing the manifest to `SimulationContext`.

### MCP Tool Signatures

- **D-04:** `simulate_domain(domain: str, scenario: str) -> dict`
  - `domain` — domain name string (e.g. `"Elevator"`). Tool looks up the bundle from `.design/bundles/<domain>.mdfbundle`.
  - `scenario` — path to a `.scenario.yaml` file.
- **D-05:** `simulate_class(class_name: str, scenario: str) -> dict`
  - `class_name` — name of the class to simulate in isolation.
  - `scenario` — path to a `.scenario.yaml` file scoped to that class.
  - Uses the full runtime (registry, scheduler, ctx) but loads only that class's generated code from its domain bundle. Relationships to other classes and bridge calls are mocked.
- **D-06:** Both tools return:
  ```python
  {
      "total_steps": int,
      "final_instance_states": {"ClassName": {"id_field": "StateName", ...}, ...},
      "errors": [{"type": str, "message": str}, ...],
      "trace_file": "path/to/trace.json"
  }
  ```
- **D-07:** Trace files written to `.design/traces/`, named `<domain>_<scenario_stem>_<timestamp>.json`.

### Scenario File Format

- **D-08:** Scenario files live in `.design/scenarios/`. Files use `.scenario.yaml` extension.
- **D-09:** Scenario YAML is validated against a **Pydantic schema** (`schema/scenario_schema.py`) before the simulation starts. Validation errors are hard failures with clear messages.
- **D-10:** No `domain` field in the scenario file — the domain is implied by the `simulate_domain()` / `simulate_class()` tool argument.
- **D-11:** Instances have a **`name` alias** used to reference them in `relationships`, `events`, and `triggers`. The `name` is local to the scenario file.
- **D-12:** Initial relationships expressed as an inline top-level `relationships` list:
  ```yaml
  relationships:
    - rel: R1
      a: elev1      # instance name alias
      b: shaft1
  ```
  Called via `ctx.relate()` during setup, before events fire.
- **D-13:** **Pre-flight multiplicity check** — before simulation starts, verify all required relationship multiplicities from the domain manifest are satisfied by the scenario's initial population. Hard fail if violated.
- **D-14:** Bridge mocks provided in a **separate companion `.mocks.yaml` file** (not inline in the scenario). The tool accepts an optional `mocks` path argument, or looks for `<scenario_stem>.mocks.yaml` alongside the scenario.
- **D-15:** **Sender is always required** for all events — scenario events, trigger-fired events, and method calls. Missing sender is a schema validation error (caught by Pydantic before simulation).

### Event Scheduling in Scenario

- **D-16:** Events support **two optional delay fields** (opt-in, mutually exclusive):
  - `at_ms: <float>` — absolute simulation time to fire this event
  - `after_ms: <float>` — delay relative to the previous event in the scenario list
  - Events with no delay field fire at t=0 (or immediately if simulation already running).
- **D-17:** **Method calls** in the scenario (`call: method_name`) execute the action body of that method directly on the target instance (with the correct `ctx`, `self_dict`, `params`). Any `generate` statements inside the action body follow normal scheduler rules. Sender is required.

### Conditional Triggers

- **D-18:** Triggers are evaluated **after each micro-step**.
- **D-19:** Trigger conditions can test **instance state** and/or **attribute values**. When both are present, logic is AND (all conditions must be true).
  ```yaml
  triggers:
    - when:
        instance: elev1
        state: Idle          # optional state condition
        attr: current_floor  # optional attribute condition
        eq: 2                # attribute equality check
      then:
        event: ServiceComplete
        target: elev1
        sender: elev1
        args: {}
      repeat: false          # default — disarms after first fire
  ```
- **D-20:** Trigger action can **post an event** (`event:`) or **call a method** (`call:`), using the same format as scenario events (sender required).
- **D-21:** `repeat: false` (default) — trigger disarms after first fire. `repeat: true` — trigger re-arms and fires every time the condition is met.

### Elevator Verification Scenario

- **D-22:** Exit gate scenario is a **multi-floor sequence with competing calls** — multiple floor requests queued, elevator services them in sequence. Exercises all active classes (Elevator, Shaft, Floor, Door, Buttons).
- **D-23:** Verification asserts **key micro-step milestones** — specific `transition_fired` micro-steps for each expected state change. Does not require exact ordering of every step. Regression-safe without being brittle.

### Claude's Discretion

- Internal module layout of the bundle loader (e.g., `engine/bundle_loader.py` vs a top-level `runner/` module)
- Engine version constant location (e.g., `engine/__init__.py`)
- How callable rebinding from generated modules works internally
- Exact Pydantic field names and structure within `scenario_schema.py`
- Temp dir management strategy (contextmanager, explicit cleanup, etc.)
- How `simulate_class` identifies which domain bundle to load from just the class name

## Existing Code Insights

### Reusable Assets
- `engine/ctx.py` — `run_simulation(domain_manifest, scenario, bridge_mocks)` is already the top-level entry point. Phase 5.3 wraps this; `load_scenario()` needs to be extended for relationships and the new scenario format.
- `engine/manifest.py` — `ClassManifest` and `TransitionEntry` TypedDicts are the target structures the bundle loader must populate with rebound callables.
- `tools/simulation.py` — Currently a stub. Phase 5.3 implements the two MCP tool functions here.
- `compiler/__init__.py` — Public `compile_model()` API is complete. Phase 5.3 does not touch this.

### Established Patterns
- Pydantic models with `populate_by_name=True` — `scenario_schema.py` should follow this pattern
- Flat uv layout — `schema/scenario_schema.py` sits alongside other schema files
- Error accumulation pattern (`ErrorAccumulator`) from `compiler/error.py` — pre-flight checks should follow same UX

### Integration Points
- `tools/simulation.py` calls into the bundle loader, which calls into `engine/ctx.py`
- Bundle loader must understand the `.mdfbundle` zip layout from `compiler/packager.py`
- `schema/scenario_schema.py` is the Pydantic schema the loader validates against before any simulation
- `.design/bundles/` — bundle lookup directory for `simulate_domain(domain_name, ...)`
- `.design/scenarios/` — scenario file location
- `.design/traces/` — trace output location

<specifics>
## Specific Ideas

- Scenario format example (elevator):
  ```yaml
  instances:
    - class: Elevator
      name: elev1
      id: {elevator_id: 1}
      state: Idle
      attrs: {current_floor: 1}
    - class: Floor
      name: floor1
      id: {floor_id: 1}
  relationships:
    - rel: R1
      a: elev1
      b: shaft1
  events:
    - event: CallButton_pressed
      target: button1
      sender: button1
      args: {floor: 2}
      at_ms: 0
    - call: press
      target: button2
      sender: button2
      after_ms: 500
  triggers:
    - when:
        instance: elev1
        state: Idle
        attr: current_floor
        eq: 2
      then:
        event: ServiceComplete
        target: elev1
        sender: elev1
      repeat: false
  ```
- The `engine/ctx.py` `load_scenario()` method needs to be extended to handle: instance aliases, relationship setup, and conditional triggers.
- The pre-flight multiplicity check should produce the same UX as `validate_model()` — named issues with location context.

</specifics>
