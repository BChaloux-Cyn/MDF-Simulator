## Warnings

### WR-01: `tmpdir` leaked on `BundleVersionError` and `BundleCorruptError`

**File:** `engine/bundle_loader.py:45-69`
**Issue:** `tmpdir` is created with `tempfile.mkdtemp` at line 45 before the zip is
inspected. If `BundleCorruptError` is raised (path traversal check, lines 53-54) or
`BundleVersionError` is raised (lines 66-69), the function exits via exception without
cleaning up `tmpdir`. On the success path the caller receives `tmpdir` and is
responsible for cleanup â€” but on the error paths the temp directory is silently
abandoned. On Windows (the active platform) this can cause "file in use" failures for
subsequent temp-dir cleanup.
**Fix:**
```python
def load_bundle(bundle_path: Path) -> tuple[dict, Path]:
    bundle_path = Path(bundle_path)
    tmpdir = Path(tempfile.mkdtemp(prefix="mdfbundle_"))
    try:
        # ... all existing extraction and validation logic ...
        return manifest, tmpdir
    except Exception:
        shutil.rmtree(tmpdir, ignore_errors=True)
        raise
```

### WR-02: Path traversal guard is bypassable on Windows due to case-sensitive `startswith`

**File:** `engine/bundle_loader.py:49-53`
**Issue:** The guard at lines 49-53 uses `str(resolved).startswith(tmpdir_resolved)`.
On Windows `Path.resolve()` may return differently-cased drive letters across calls
(e.g., `C:\Users\...` vs `c:\Users\...`), so the check can produce false negatives.
Additionally, if `tmpdir_resolved` does not end with a path separator, a crafted path
whose resolved string starts with the same prefix but points outside the directory
could pass.
**Fix:**
```python
import os
tmpdir_prefix = str(tmpdir.resolve()).lower() + os.sep
for name in zf.namelist():
    resolved = str((tmpdir / name).resolve()).lower()
    if not resolved.startswith(tmpdir_prefix):
        raise BundleCorruptError(f"Bundle contains unsafe path: {name!r}")
```

### WR-03: Transform failures are silently swallowed â€” malformed action bodies produce silent no-ops

**File:** `compiler/codegen.py:237-244`
**Issue:** `transform_action` at line 237 is wrapped in a bare `except Exception` that
sets `transformed = ""`. An empty body becomes `pass` in the generated action
function, with no `CompileError` accumulated and no `CompilationFailed` raised. A
model with a malformed action body compiles to a valid bundle that silently no-ops
every affected action. The same pattern applies to `transform_guard` at lines 277-283.
`test_codegen_fixes.py:test_compile_failure_emits_warning` only verifies that a
`UserWarning` is emitted (which it is, via `warnings.warn`), but does not verify that
compilation of the full bundle ultimately fails.
**Fix:** Either accumulate a `CompileError` and surface it via `CompilationFailed`, or
at minimum document the design decision that malformed bodies intentionally degrade to
`pass` rather than failing the compile. The current behaviour is inconsistent with the
accumulator pattern used in `compiler/__init__.py:110-115`.

### WR-04: `test_run_scenario_yields_micro_steps_and_resolves_aliases` uses wrong dict key â€” transition never fires

**File:** `tests/test_simulation_runner.py:483-488`
**Issue:** The manually constructed transition table entry uses:
```python
manifest["class_defs"]["Elevator"]["transition_table"] = {
    ("Idle", "Floor_assigned"): [{"next_state": "Departing", "action_fn": None, "guard_fn": None}],
}
```
This is correct â€” but earlier in the test file (line 483 shows the entry as a bare
dict, not a list). In the actual file at lines 483-486 the value is not wrapped in a
list:
```python
("Idle", "Floor_assigned"): {"next_state": "Departing", "action_fn": None, "guard_fn": None}
```
The scheduler reads `ttable.get((curr_state, event.event_type))` and expects a
`list[TransitionEntry]`. If a bare dict is returned, iterating over it yields its keys
(`"next_state"`, `"action_fn"`, `"guard_fn"`) rather than entry dicts. The guard-chain
loop at lines 414-440 of `scheduler.py` will attempt to call `candidate.get("guard_fn")`
on a string, which will raise `AttributeError` â€” but this is inside a try/except that
yields `ErrorMicroStep`. The test only asserts `len(steps) > 0`, which passes because
`EventReceived` is emitted before the error. The transition is never confirmed to fire.

Additionally the `_make_minimal_manifest` helper on line 312 also leaves
`"transition_table": {}` for the base manifest, which is then overwritten â€” the
helper's default doesn't cause the issue but the per-test override must be a list.

**Fix:** Wrap the entry in a list (matching the D-06 list-format convention) and add a
`TransitionFired` assertion:
```python
manifest["class_defs"]["Elevator"]["transition_table"] = {
    ("Idle", "Floor_assigned"): [
        {"next_state": "Departing", "action_fn": None, "guard_fn": None}
    ],
}
# ...
from engine.microstep import TransitionFired
fired = [s for s in steps if isinstance(s, TransitionFired)]
assert len(fired) == 1
assert fired[0].to_state == "Departing"
```

### IN-01: `_FIXED_DATE` constant is defined but never used in `codegen.py`

**File:** `compiler/codegen.py:38`
**Issue:** `_FIXED_DATE = (2020, 1, 1, 0, 0, 0)` is declared with a comment "used for
reproducible zip timestamps elsewhere". It is not referenced anywhere in `codegen.py`.
If the packager no longer imports it from this module, this is dead code.
**Fix:** Remove the constant from `codegen.py`, or move it to `packager.py` where it
is actually consumed.

### IN-02: `parser` parameter accepted by `generate_class_module` is unused

**File:** `compiler/codegen.py:185`
**Issue:** The `parser` parameter is documented as "unused directly here â€”
transform_action/transform_guard build their own". Accepting an unused parameter in the
public API is misleading and forces callers to provide and maintain an argument that
has no effect.
**Fix:** Remove `parser` from the signature and update callers (`compiler/__init__.py:106`
and test helpers that pass `STATEMENT_PARSER` or `None`).

### IN-03: Local `TransitionEntry` TypedDict in test file shadows the engine type

**File:** `tests/test_compiler_manifest.py:550-555`
**Issue:** A local `TransitionEntry` TypedDict is defined at module scope after all
imports. Its field types (`action_fn: None`, `guard_fn: None`) differ from the real
`engine.manifest.TransitionEntry` (`Callable | None`). It is only used for a single
structural assertion and causes no runtime bug, but any future test that references
`TransitionEntry` at this module scope will silently get the wrong type.
**Fix:** Remove the local definition and import `TransitionEntry` from `engine.manifest`
at the top of the file, matching the pattern in other test files.
