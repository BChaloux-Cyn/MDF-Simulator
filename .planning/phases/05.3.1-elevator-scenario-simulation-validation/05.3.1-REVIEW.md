Reviewed the compiler pipeline (codegen, manifest builder, `__init__.py`), the engine

## Warnings

### WR-01: `tmpdir` leaked on `BundleVersionError` and `BundleCorruptError`

**File:** `engine/bundle_loader.py:45-69`

**Issue:** `tmpdir` is created with `tempfile.mkdtemp` at line 45 before the zip is
inspected. If `BundleCorruptError` is raised (path traversal check, lines 53-54) or
`BundleVersionError` is raised (line 66-69), the function exits via exception without
cleaning up `tmpdir`. The calling code (e.g., `test_simulation_runner.py` wraps
`load_bundle` in a `try/finally` with `shutil.rmtree`, but only for the happy path).
On the error paths the temp directory is abandoned on disk.

On Windows (the active platform) this also causes "file in use" failures when a
subsequent test tries to clean the session temp root, matching Pitfall 5 noted in
`RESEARCH.md`.

**Fix:**
```python
def load_bundle(bundle_path: Path) -> tuple[dict, Path]:
    bundle_path = Path(bundle_path)
    tmpdir = Path(tempfile.mkdtemp(prefix="mdfbundle_"))
    try:
        with zipfile.ZipFile(bundle_path) as zf:
            tmpdir_resolved = str(tmpdir.resolve())
            for name in zf.namelist():
                resolved = (tmpdir / name).resolve()
                if not str(resolved).startswith(tmpdir_resolved):
                    raise BundleCorruptError(f"Bundle contains unsafe path: {name!r}")
            zf.extractall(tmpdir)
        # ... remainder of function ...
        return manifest, tmpdir
    except Exception:
        shutil.rmtree(tmpdir, ignore_errors=True)
        raise
```

### WR-02: `_process_creation` does not yield `TransitionFired` when initial state is set

**File:** `engine/scheduler.py:475-519`

**Issue:** `_process_creation` sets the instance's initial state (line 491) and yields
`EventReceived`, but it never yields `TransitionFired`. The `execute` loop in
`test_engine.py:test_creation_event_processing` only checks that `get_state` returns
`"Idle"` after processing — it does not assert a `TransitionFired` step. Consumers of
the trace (e.g., `test_elevator_scenarios.py:_get_transitions`) filter on
`type == "transition_fired"` and will silently miss the initial-state entry.

Whether this is intentional (initial placement is not a transition) should be explicit
in the design docs. If it is intentional, a comment is needed. If it is not, a
`TransitionFired(from_state="", to_state=initial_state, ...)` should be yielded at
line 492.

**Fix (if creation should be recorded as a transition):**
```python
if initial_state is not None:
    self._registry.set_state(target_class, target_id, initial_state)
    yield TransitionFired(
        from_state="",
        to_state=initial_state,
        class_name=target_class,
        instance_id=dict(target_id),
    )
```

### WR-03: Bare `except Exception` in `generate_class_module` silently swallows transform errors

**File:** `compiler/codegen.py:229-231`

**Issue:** The `transform_action` call at line 229 is wrapped in a bare
`except Exception: transformed = ""`. A transform error produces an empty body, which
silently becomes `pass` in the generated action function. There is no log, no
`CompileError`, and no accumulator entry. A model with a malformed action body will
compile and produce a bundle that silently no-ops every action it contains.

The same pattern appears at line 257-259 for `transform_guard`.

**Fix:** Either accumulate a `CompileError` for the malformed action body or re-raise
after logging. Minimal fix:
```python
try:
    transformed = transform_action(action_src, source_file, 0)
except Exception as exc:
    acc = ErrorAccumulator()  # or pass an accumulator in
    acc.add(CompileError(
        file=source_file,
        line=0,
        message=f"transform_action failed for state {state_name!r}: {exc}",
    ))
    acc.raise_if_any()
```

### WR-04: `test_run_scenario_yields_micro_steps_and_resolves_aliases` uses wrong `TransitionEntry` key

**File:** `tests/test_simulation_runner.py:483-488`

**Issue:** The manually constructed transition table entry at lines 483-488 uses
`"to_state"` as the key:
```python
manifest["class_defs"]["Elevator"]["transition_table"] = {
    ("Idle", "Floor_assigned"): {
        "to_state": "Departing",   # WRONG key
        "action_fn": None,
        "guard_fn": None,
    }
}
```

The scheduler reads `entry.get("next_state")` (see `scheduler.py:406`). The key
`"to_state"` is silently ignored, so `next_state` is `None`. The event-ignored
short-circuit at line 411 (`next_state is None and action_fn is None and guard_fn is
None`) then silently consumes the event without firing the transition. The test only
asserts `len(steps) > 0` (which passes because `EventReceived` and
`SchedulerSelected` steps are still emitted) and does not verify the transition
actually fired. The test does not catch the bug it is supposed to test.

**Fix:** Use the correct key `"next_state"` and add a `TransitionFired` assertion:
```python
manifest["class_defs"]["Elevator"]["transition_table"] = {
    ("Idle", "Floor_assigned"): {
        "next_state": "Departing",  # correct key
        "action_fn": None,
        "guard_fn": None,
    }
}
# ... after run_scenario ...
transitions = [s for s in steps if hasattr(s, "type") and s.type == "transition_fired"]
assert len(transitions) == 1
assert transitions[0].to_state == "Departing"
```

### IN-03: `is_abstract` is always `False` in `build_class_manifest`

**File:** `compiler/manifest_builder.py:174`

**Issue:** `is_abstract = False  # canonical doesn't expose abstract; default false`
is hardcoded. The `ClassManifest` TypedDict declares `is_abstract: bool` and
`test_engine.py:test_registry_supertype_inheritance` demonstrates that
`is_abstract: True` is needed to reject abstract instantiation. The registry uses
`is_abstract` at runtime — if the compiler always emits `False`, abstract supertypes
in the elevator domain would be instantiable, which is incorrect xUML semantics.

This is an existing gap rather than a new regression, but it is worth tracking.

**Fix:** Derive `is_abstract` from the class stereotype:
```python
is_abstract = (entry.stereotype == "abstract")
```
or expose an `is_abstract` flag in `CanonicalClassEntry` and propagate it.

### IN-05: `test_all_scenarios_no_error_microsteps` re-runs all 5 scenarios redundantly

**File:** `tests/test_elevator_scenarios.py:229-246`

**Issue:** This test executes all five scenarios again through `_run_scenario`, each of
which compiles and runs the full simulation. The individual scenario tests
(`test_scenario_1_*` through `test_scenario_5_*`) already assert `trace_errors == []`.
This omnibus test provides no additional coverage and doubles execution time for the
scenario suite, which is especially costly given the module-scoped `elevator_runtime`
fixture compiles the model once but each `_run_scenario` call runs the full
simulation.

**Fix:** Either remove `test_all_scenarios_no_error_microsteps` or restructure it to
reuse results already computed by the individual tests (via a session-scoped results
fixture) rather than re-running them.
