**Step 1: Diagnose exact failures**

Before fixing anything, manually test each body through `transform_action` to identify the exact exception. Run in Python:

**Step 2: Fix identified gaps**

Based on research, the likely gaps are:

a) **`generate Event to <set_variable>`**: The `generate_stmt` normalizes target using `_normalize_target` which calls `s.isidentifier()`. A set variable IS an identifier, so this should work. BUT generating to a set means generating one event per instance in the set. The transformer must detect set targets and emit a for-loop:
```python
# In generate_stmt, after determining the target:
# If the target variable was populated via select_many, the generated code
# must iterate. Since we can't statically determine if a var is a set,
# emit ctx.generate_to_set() which handles both single instances and sets.
# Actually: ctx.generate already handles single targets. For sets, we need:
for _inst in target_var:
    ctx.generate("Event", target=_inst["__instance_key__"], args={...})
```

Add a `generate_to_set` pattern: when the target identifier was populated by `select_many`, the action body already has it as a list. The simplest approach: in `generate_stmt`, when the target is a plain NAME that is NOT "self", emit:
```python
ctx.generate_to_set("{event_name}", instances={target_var}, args={params_dict})
```
And add `generate_to_set` to `engine/ctx.py` (or use a for-loop in the generated code).

HOWEVER: the simpler approach (and more correct for xUML semantics) is to emit a for-loop in the generated code:
```python
for _tgt in {target_var}:
    ctx.generate("{event_name}", target=_tgt["__instance_key__"], args={params_dict})
```

But we can't distinguish set variables from single-instance variables at compile time. The safest approach: always wrap non-self targets in a helper that handles both:
- If target is a dict (single instance): generate to it
- If target is a list (set of instances): iterate and generate to each

Add to generated code preamble or as a ctx helper method.

b) **`method_call` on variables (not self)**: `sf.has_value()` and `sf.value()` — the `method_call` transformer handles `obj != "self"` by emitting `obj.method()`. For Optional types from `select any`, `has_value()` should map to `obj is not None` and `value()` should be identity (the value itself). The transformer should special-case these Optional methods:
- `var.has_value()` -> `(var is not None)`
- `var.value()` -> `var`

In `method_call` and `method_call_stmt`:
```python
if method == "has_value":
    return f"({obj} is not None)"
if method == "value":
    return obj
```

c) **Lambda bodies containing statements**: The `lambda_expr` transformer currently emits `lambda params: body` which only works for single-expression bodies. For multi-statement lambda bodies (used in `filter`, `flat_map`, `sort`), emit a closure instead. Check if the body contains newlines or multiple statements:
```python
if "\n" in body or ";" in body:
    # Multi-statement lambda -> emit def + call
    return f"(lambda {params_str}: {body})"  # This won't work for statements
```

Actually, the lambda bodies in the elevator model are all single-expression boolean returns (used in `where`, `filter`, `sort`). Verify this. If the body is a single expression returning a boolean, the current `lambda params: body` form works. The issue may be elsewhere.

d) **Chained method calls with lambda arguments**: `set.filter(lambda_expr)`, `set.flat_map(lambda_expr)`. The `chained_method_call` transformer emits `chain.method(args)`. The lambda_expr child is already transformed to a Python lambda string. This should work if the grammar parses the lambda inside the method call parentheses. Verify that the grammar's `arglist` rule accepts `lambda_expr` as an argument.

If the grammar doesn't allow lambda_expr inside arglist (likely the issue), the grammar needs a new production:
```
arglist: expr ("," expr)*
       | lambda_expr
```
Or add `lambda_expr` as an atom/primary expression in the grammar.

e) **For Arriving entry**: The body uses `self._get_lit_buttons()` which is a method call on self. This becomes `self_dict._get_lit_buttons()` but `self_dict` is a plain dict — it has no methods. This is a **model-level method** that needs to be compiled separately and injected. For scenarios 2-3, a simplified stub that just does `generate Arrived to self;` is sufficient per CONTEXT.md. For scenario 5, the full body is needed.

**Pragmatic approach for Arriving entry**: The `_get_lit_buttons()` method body would need to be compiled and attached to the instance dict or available as a standalone function. This is a significant architectural addition. Per CONTEXT.md "Claude's Discretion" section, minimal stub implementations for complex bodies are acceptable as intermediate steps.

For Arriving entry action, replace the complex body with a simplified version that:
1. Sets `self.next_stop_floor = self.current_floor` (simple default — stay at current floor)
2. Generates `Arrived` to self

This can be done by editing the model YAML OR by adding a compile-time fallback in codegen that detects when the full body fails and substitutes a known-safe stub. The YAML edit is cleaner and more transparent.

Actually, per CONTEXT.md locked decisions: "Fix root causes in compiler/codegen.py and compiler/transformer.py — not workarounds". So we should fix the transformer, not stub the model.
