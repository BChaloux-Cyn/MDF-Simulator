## Summary

This phase fixes two categories of compiler bugs and fills in missing action body compilation so that five elevator scenarios execute end-to-end. The work is almost entirely in `compiler/codegen.py` (action-assignment direction) and `compiler/transformer.py` (missing grammar constructs). The engine itself needs no changes.

**The core defect:** `codegen.py` assigns each state's entry action to all outgoing transitions from that state. xUML semantics require the entry action to fire on transitions *into* the state — so the action should be assigned to transitions whose `next_state` equals the state name.

**Primary recommendation:** Fix the action-assignment direction bug first (it affects all classes), then fix enum scoping, then compile missing action bodies in order of scenario dependency (Lit → Arriving → Exchanging). Scenario YAML files should be authored alongside each transformer fix.

## Action-Assignment Direction Bug (Issue #1)

### Root Cause (VERIFIED: codegen.py lines 220-222)

```python
# CURRENT (WRONG): assigns action_<State>_entry to outgoing transitions
for (s, e) in transition_table:
    if s == state_name:
        action_fn_map[(s, e)] = fn_name
```

In xUML/Shlaer-Mellor, a state's entry action fires when the state is *entered* — i.e., when a transition fires whose `next_state` is this state. The fix maps each `action_<State>_entry` to transitions where `transition_table[(s,e)]["next_state"] == state_name`.

```python
# CORRECT: assigns action_<State>_entry to incoming transitions
for (s, e), cell in transition_table.items():
    if cell["next_state"] == state_name:
        action_fn_map[(s, e)] = fn_name
```

### Impact on Every Generated Class (VERIFIED: inspection of all generated .py files)

| Class | Wrong assignment example | Correct assignment |
|-------|--------------------------|-------------------|
| CarDoor | `action_Closed_entry` on `(Closed, Open_cmd)` (outgoing) | Should be on `(Closing, Closed_now)` (incoming to Closed) |
| CarDoor | `action_Opening_entry` on `(Opening, Opened)` (outgoing) | Should be on `(Closed, Open_cmd)` (incoming to Opening) |
| ShaftDoor | Same pattern — all actions assigned to outgoing transitions | Should all be on incoming transitions |
| Elevator | `action_Departing_entry` on `(Departing, Ready)` — uses `params["floor_num"]` but `Ready` has no params | Should be on `(Idle, Floor_assigned)` — which carries `floor_num` |
| FloorCallButton | `action_Lit_entry` on `(Lit, Call_served)` — outgoing | Should be on `(Idle, Activated)` — incoming to Lit |

### Params Availability Rule (ASSUMED from xUML semantics)

With the corrected direction, `params` in an entry action contains the event parameters of the *triggering* event — the event that caused the transition into this state. This is correct: `Departing` is entered via `Floor_assigned(floor_num: N)`, so `params["floor_num"]` is valid in `action_Departing_entry`.

### No `params` for Self-Generated Events (VERIFIED: model + scheduler code)

States entered by a self-generated parameterless event (`Ready`, `Arrived`, `At_Destination`, `Continue`) have entry actions that must NOT read `params["x"]`. The Departing, Floor_Updating, Arriving, and Exchanging entry actions must each be checked against the triggering event's param list.

## Enum Scoping Bug (Issue #2)

### Root Cause (VERIFIED: codegen.py + generated Elevator.py)

`generate_class_module` only emits enum class definitions for types that appear in the class's *attribute* declarations (lines 183-196). `action_Departing_entry` references `Up` and `Down` (the `Direction` enum), but `Direction` appears as the type of `Elevator.direction` — an attribute of `Elevator` itself. So `Direction` **is** in `used_types` and should already be emitted.

On inspection of the generated `Elevator.py`, there is no `Direction` class present. The reason: `generate_class_module` only emits from `type_registry`, which is built from the domain `types.yaml`. The `type_registry` must be populated with `Direction` (with `kind: "enum"`, `members: ["Down", "None", "Up"]`) and `Elevator.py` must declare it.

Verification: `types.yaml` defines `Direction` as base: enum with values `Up`, `Down`, `None`. The `type_registry` emitter in `codegen.py` checks `info.get("kind") == "enum"` and emits if the type is in `used_types`. If `Direction` is in `Elevator.attributes`, it must be in `used_types`. This should already work — but the generated file shows no enum. Root cause likely in how `type_registry` is built from `types.yaml` and passed to `generate_class_module`. **Needs direct test to confirm the gap.**

Also: `None` is a Python keyword — the enum member `None` in `Direction` will conflict. Must be emitted as `Null` or quoted. [ASSUMED — needs verification]

## Missing Transition: Exchanging → Departing (Issue #6 — unlisted but critical for Scenarios 4-5)

### Root Cause (VERIFIED: manifest_builder.py + generated Elevator.py)

`_build_transition_table` uses a plain dict with `(from_state, event)` as key. When two transitions share the same `(from_state, event)` pair (like both `Exchanging → Departing` and `Exchanging → Idle` on `Door_closed`), the second one silently overwrites the first.

The model has:
```yaml
- from: Exchanging, to: Departing, event: Door_closed, guard: "self.next_stop_floor != self.current_floor"
- from: Exchanging, to: Idle,      event: Door_closed, guard: "self.next_stop_floor == self.current_floor"
```

The generated `Elevator.py` only has `("Exchanging", "Door_closed") → Idle`. The `Departing` branch is lost.

**Fix required in `manifest_builder._build_transition_table`:** Guarded sibling transitions with the same `(from_state, event)` must be stored as a list, OR the transition table format must accommodate multiple next-states with guards. The scheduler's dispatch logic must also be able to evaluate guards and select the correct branch.

**Scheduler guard evaluation (VERIFIED: engine/scheduler.py):** The scheduler already reads `guard_fn` from the transition cell. But the current table format `dict[(state, event)]` stores one cell per `(state, event)` — it cannot represent two cells for the same key.

**Resolution approach:** Change transition table format from `dict[(state, event) → cell]` to `dict[(state, event) → list[cell]]` where each cell has `{next_state, action_fn, guard_fn}`. The scheduler evaluates guards in order and picks the first that returns True. This is a schema-level change affecting `manifest_builder`, `codegen`, and `scheduler`.

**Alternative (simpler) approach:** Keep the format as-is but add a `guard_chain` structure alongside, or model the fallback transition as the unguarded default. Given that both branches are guarded (complementary guards), a simpler approach is to store one row as the "primary" and encode the alternative as a separate lookup. However, changing to `list[cell]` is cleaner and future-proof.

**Impact assessment:** This is a significant structural change. However, Scenarios 1–3 do not require it (they don't hit the `Exchanging → Departing` branch). Scenarios 4–5 do. The planner should consider whether to include this fix in Phase 05.3.1 or defer to a follow-on phase. The CONTEXT.md explicitly lists Scenario 5 (Passenger Round-Trip) as required, which does need `Exchanging → Departing`.

## Missing Action Body Compilation (Issues #3, #4, #5)

### Issue #3: `action_Lit_entry` (FloorCallButton) — `pass`

**Source pycca:**
```
Floor floor = self->R6;
Optional<ShaftFloor> sf = select any related by floor->R3;
if (sf.has_value()) {
    ShaftFloor shaft_floor = sf.value();
    Elevator elev = shaft_floor->R2->R1;
    if (elev.curr_state == Idle) {
        generate Floor_assigned(floor_num: floor.floor_num) to elev;
    }
}
```

**Why it's `pass`:** `select any related by floor->R3` uses a non-self traversal source (`floor->R3`), which goes through `traversal_chain`. The grammar and `traversal_chain` transformer should handle `floor->R3`. Let me verify.

Actually: the `select_related_stmt` rule is:
```
"select" ("any"|"many") NAME "related" "by" traversal_chain ";"
```
And `traversal_chain` is `NAME ("->" NAME)+`. So `floor->R3` parses to `traversal_chain(floor, R3)` and becomes `ctx.traverse(floor, ["R3"])`. Then `select_related_stmt` wraps it as `sf = ctx.select_any_related(floor, ["R3"])`.

The issue is the **exception handler** in `generate_class_module` (lines 210-213):
```python
try:
    transformed = transform_action(action_src, source_file, 0)
except Exception:
    transformed = ""
```
Any parse or transform failure is silently swallowed, emitting `pass`. The failing construct here may be `sf.has_value()` or `sf.value()` — these are chained method calls that the transformer may not handle (or grammar may not parse).

**Verification needed:** Run `transform_action` on this body manually to see the actual exception.

**Likely gap:** `Optional<ShaftFloor>` as a typed var decl is fine (generic_type). But `sf.has_value()` is a `method_call_stmt` on a variable (not `self` or a named class). The `method_call` transformer returns `self_dict.method()` when obj == "self", otherwise `obj.method()`. For `sf.has_value()`, obj is `sf`, which is fine. But `sf.value()` returns the instance dict — and then `shaft_floor->R2->R1` starts a traversal with `shaft_floor` as the source. This should work.

The real failure may be in `chained_attr_access` used for `elev.curr_state` — `elev` is a dict, so `elev.curr_state` should become `elev["curr_state"]` via `dotted_name`. But `dotted_name` only handles `self.x` and `rcvd_evt.x` — any other `obj.attr` falls through to `f'{obj}["{attr}"]'`. That should work.

**Most likely root cause:** Exception is thrown during Lark parse, not transform — possibly a grammar rule that doesn't cover `Optional<ShaftFloor>` in typed_var_decl position, or an ambiguity with `select any related by floor->R3`. Needs direct investigation.

### Issue #4: `action_Arriving_entry` — `pass`

**Complex constructs used:**
- `self._get_lit_buttons()` — method call on self returning a Set
- `shaft_floors.flat_map([...] |sf| -> Set<FloorCallButton> {...})` — chained method call with lambda
- `lit_calls.sort([...] |a, b| -> Boolean {...})` — sort with two-param lambda
- `sorted_calls.peek_front()` — chained method call
- `earliest->R6` — traversal from a plain variable (not self)
- `select many related by f->R6 where [...] |btn| -> Boolean {...}` inside a lambda body — nested select in lambda

**Why it's `pass`:** Almost certainly fails to parse due to `flat_map` being a chained method call with a lambda argument, or the nested `select` inside a lambda body (which would require the grammar to allow `select` in expression position inside a lambda). The `lambda_expr` rule expects `statement+` in its body — `select many related by f->R6 where [...]` is a `select_expr` (expression form), not a `select_related_stmt` (statement form).

**Key insight:** The Arriving entry action is algorithmically complex. For Scenarios 2–3, a simplified stub that just does `generate Arrived to self` is sufficient (the Arriving action is only about deciding the next stop — if we set `next_stop_floor` correctly before running, the stub just needs to emit `Arrived`). The CONTEXT.md notes this as an option.

**Decision for planner:** For Scenarios 2-3, a minimal stub is acceptable. For Scenario 5 (full passenger round-trip), the full body is needed. The planner should stage this.

### Issue #5: `action_Exchanging_entry` — `pass`

**Complex constructs used:**
- `select many related by self->R4 where [self.current_floor] |btn| -> Boolean {...}` — select-with-lambda where clause
- `generate Floor_served to dest_buttons` — generate to a Set (generates one event per instance in set)
- `shaft_floors.filter([self.current_floor] |s| -> Boolean {...})` — chained method call with lambda
- `current_sf.peek_front()` — chain
- `select many related by floor->R6 where [...] |btn| -> Boolean {...}` — non-self traversal source
- `generate Call_served to call_buttons` — generate to a Set
- `CarDoor car_door = self->R7` — typed var decl with traversal RHS
- `generate Open_cmd to car_door` — generate to instance

**Key insight:** For Scenarios 4-5, the minimum needed from Exchanging entry is `generate Open_cmd to car_door`. The rest (notifying buttons) can be a minimal stub for initial scenarios.

## Code Generation Fix Plan (Ordered by Dependency)

### Fix 1: Action-assignment direction (codegen.py)

Change `action_fn_map` population from "outgoing from state_name" to "incoming to state_name":

```python
# Replace lines 220-222:
for (s, e), cell in transition_table.items():
    if cell["next_state"] == state_name:
        action_fn_map[(s, e)] = fn_name
```

**Impact:** ALL generated files change. Existing test `test_elevator_compiles_and_runs_multi_floor_scenario` may break because action functions are now assigned to different transitions. The existing scenario intentionally starts in `Arriving` and uses empty-body actions — after the fix, actions on `(Arriving, Arrived)` will be `action_Exchanging_entry` (incoming to Exchanging), not `action_Arriving_entry`. This changes which function runs.

**Existing scenario compatibility:** The existing `elevator_multi_floor.scenario.yaml` starts the elevator in `Arriving` and sends `Arrived` and `Door_closed`. After the fix:
- `(Arriving, Arrived) → Exchanging` will run `action_Exchanging_entry` (which is `pass`)
- `(Exchanging, Door_closed) → Idle` will run `action_Idle_entry` (which is `pass`)
Both are still `pass`, so the existing test should still pass.

### Fix 2: Enum scoping (codegen.py)

Add logic to scan action body source text for enum member names that are NOT in attribute types but ARE used in action code. Alternative: always emit all domain enums in every class module. The simplest fix: emit ALL types from type_registry into every generated module (not just those in attribute types). This is safe since the types module is a closed set.

**Simpler fix:** Pass the domain's `types.yaml` enum definitions through to all class modules as a domain-level import or inline declaration. The cleanest approach is to add a `domain_types` block to the generated module that emits all enums/typedefs from the type_registry unconditionally.

### Fix 3: Guard sibling transitions (manifest_builder.py + codegen.py + scheduler.py)

The transition table format must change to support multiple guarded branches per `(from_state, event)` pair.

**Option A (recommended):** Change value type from `TransitionEntry` to `list[TransitionEntry]`. All downstream consumers (codegen, scheduler) must handle lists. The scheduler evaluates guards in list order; the first guard returning `True` wins; if no guard matches and no unguarded fallback exists, emit `ErrorMicroStep`.

**Option B:** Add a separate `guard_sibling` dict alongside the main transition table for overflow branches. More targeted but messier.

**Scope:** Option A touches `engine/manifest.py` (type signature), `compiler/manifest_builder.py` (builder logic), `compiler/codegen.py` (transition table renderer), `engine/scheduler.py` (dispatch logic). This is significant but straightforward.

## Common Pitfalls

### Pitfall 1: Guard-Only Transition Cell Is Not Enough

The Exchanging → Departing branch is missing from the transition table entirely (key collision in dict). Adding just the guard to the existing Idle branch will not fix Scenario 5 — the table structure must change.

**How to avoid:** Verify the manifest JSON for the compiled bundle shows both Door_closed branches for Exchanging.

### Pitfall 2: `params` Is Wrong for the New Action Assignment

After the direction fix, `action_Departing_entry` runs on `(Idle, Floor_assigned)`. The `params` dict for `Floor_assigned` contains `{"floor_num": N}`. The action uses `params["floor_num"]` — this is correct. But every other action must also be checked: does the event whose firing causes entry into this state carry the params the action expects?

**How to avoid:** For each entry action, look up the triggering event(s) (the transitions whose next_state equals the action's state) and verify param compatibility.

### Pitfall 3: The Existing E2E Scenario May Break After Direction Fix

The existing `elevator_multi_floor.scenario.yaml` starts in `Arriving`, sends `Arrived`. Before fix: `(Arriving, Arrived)` runs `action_Arriving_entry` (pass). After fix: `(Arriving, Arrived)` runs `action_Exchanging_entry` (pass). Both are pass, so the test still passes — but this should be explicitly verified.

**How to avoid:** Run the full 532-test suite after each compiler fix before proceeding.

### Pitfall 4: Silent `except Exception: pass` in codegen.py Hides Real Failures

The exception swallow at lines 210-213 of `codegen.py` silently turns compile errors into `pass` bodies. During development, replace with logging or re-raise. Remove the swallow (or convert to a warning) so compiler failures are visible.

**How to avoid:** Add test assertions that verify specific action functions are NOT `pass` for critical bodies.

### Pitfall 5: `None` as Enum Member Name

`Direction` has `None` as a member, which conflicts with Python's built-in `None`. The generated enum will fail:
```python
class Direction(enum.Enum):
    Down = 'Down'
    None = 'None'   # SyntaxError
    Up = 'Up'
```

**How to avoid:** Map `None` → `Null` (or `Nothing`) in the enum renderer, or quote it specially. This must be handled in `_render_enum`.

### Pitfall 6: Recompile vs Use Extracted Bundle

The extracted bundle files in `examples/elevator/.design/bundles/extracted/` are pre-built and not auto-rebuilt. Tests that call `compile_model` (via the `elevator_runtime` fixture) recompile from source — so they will pick up codegen fixes. But the static extracted files must be manually rebuilt. Don't confuse the static extracted outputs with the test-compiled outputs.
