# Phase 05.3.1 Context: Elevator Scenario Simulation Validation

## Goal

Fix the compiler pipeline and action body compilation so that five elevator
scenarios execute correctly end-to-end in the simulation engine. Passing
criteria: all code generates correctly (no stub `pass` bodies where behavior
is required) and all five scenarios produce the expected micro-step stream
without errors.

## Background

After fixing the `guard_fn` pipeline bug (Phase 05.3.1 trigger), we attempted
to write a "user pushes button Ôćĺ elevator travels Ôćĺ door opens" scenario. This
exposed two categories of problems in the compiled output:

1. **Codegen action-assignment direction bug** ÔÇö `codegen.py` maps each
   state's entry action to transitions *leaving* that state (outgoing). In
   xUML/Shlaer-Mellor semantics, entry actions fire on *entry* to the state ÔÇö
   so they should be associated with *incoming* transitions (the transition
   whose firing causes the state to be entered), not outgoing ones.

   Concrete impact: `action_Departing_entry` is assigned to `(Departing,
   Ready) Ôćĺ Moving`. That action uses `params["floor_num"]`, which is a param
   of `Floor_assigned` (the event that triggered entry into Departing), not
   `Ready` (which has no params). Running this causes a `KeyError`.

2. **Complex action bodies compiled to `pass`** ÔÇö Several state entry actions
   use navigation patterns (`select many related by`, `self._get_lit_buttons()`,
   set operations) that the transformer does not yet handle. These are silently
   emitted as `pass`.

3. **Enum scoping in generated files** ÔÇö `action_Departing_entry` references
   `Up` and `Down` (the `Direction` enum), but the generated `Elevator.py`
   does not import or define them.

## What Currently Works (pre-fix)

- `Arriving Ôćĺ Exchanging` (action is `pass` ÔÇö no crash)
- `Exchanging Ôćĺ Idle` via `Door_closed` (guard now compiled correctly after
  guard_fn fix; action is `pass`)
- `Floor_Updating` self-event logic (`At_Destination` / `Continue`) ÔÇö action
  body fully compiled
- `CarDoor.Closed_now Ôćĺ Closed` entry action: navigates R7 to elevator,
  sends `Door_closed` ÔÇö compiled correctly via `ctx.traverse`
- `ShaftDoor` full door cycle (Open_cmd Ôćĺ Opening Ôćĺ Open Ôćĺ Closing Ôćĺ Closed)
  ÔÇö action bodies compiled, uses `ctx.generate` with `delay_ms`

## What Needs to Be Fixed

| # | Issue | Affected Files |
|---|-------|---------------|
| 1 | Codegen action-assignment maps entry actions to outgoing transitions; should map to incoming (destination state's action fires with triggering event's params) | `compiler/codegen.py` |
| 2 | `Up`/`Down` enum not in scope in generated `Elevator.py` | `compiler/codegen.py` (type_registry emission) |
| 3 | `action_Lit_entry` (FloorCallButton) is `pass` ÔÇö navigation to elevator and `Floor_assigned` generate not compiled | `compiler/transformer.py` |
| 4 | `action_Arriving_entry` is `pass` ÔÇö complex navigation + method calls (`_get_lit_buttons`, `_pick_next_stop`, `flat_map`, `sort`, `peek_front`) | `compiler/transformer.py` |
| 5 | `action_Exchanging_entry` is `pass` ÔÇö navigates to DestFloorButtons + FloorCallButtons + CarDoor, generates to sets | `compiler/transformer.py` |

## Scenarios

### Scenario 1: Door Open/Close Cycle
Elevator already at destination (Idle or Exchanging). An `Open_cmd` is sent
directly to CarDoor.

Expected path:
- CarDoor: Closed Ôćĺ Opening (action: send `Open_cmd` to ShaftDoor via R7ÔćĺR1ÔćĺR2ÔćĺR8)
- ShaftDoor: Closed Ôćĺ Opening (action: delay `Done_opening` to self)
- ShaftDoor: Opening Ôćĺ Open (action: send `Opened` to CarDoor via R8ÔćĺR2ÔćĺR1ÔćĺR7)
- CarDoor: Opening Ôćĺ Open (action: delay `Close_cmd` to self, 5s)
- CarDoor: Open Ôćĺ Closing (action: send `Close_cmd` to ShaftDoor)
- ShaftDoor: Open Ôćĺ Closing (action: delay `Done_closing` to self)
- ShaftDoor: Closing Ôćĺ Closed (action: send `Closed_now` to CarDoor)
- CarDoor: Closing Ôćĺ Closed (action: send `Door_closed` to Elevator via R7)
- Elevator: Idle receives `Door_closed` Ôćĺ stays Idle (event_ignored cell)

Validates: compiled traversal actions, delay events, cross-instance event chains.
Does NOT require the action-assignment fix or complex navigation compilation.

### Scenario 2: Direct Floor Travel (No Button)
Inject `Floor_assigned(floor_num: 3)` directly to Elevator (bypassing
FloorCallButton). Elevator starts at floor 1, Idle.

Expected path:
- Elevator: Idle Ôćĺ Departing (`Floor_assigned`, action fires with floor_num=3)
- Elevator: Departing Ôćĺ Moving (self-generates `Ready` in Departing entry action)
- Inject `Floor_reached(floor_num: 3)` externally
- Elevator: Moving Ôćĺ Floor_Updating
- Elevator: Floor_Updating Ôćĺ Arriving (self-generates `At_Destination`)
- Elevator: Arriving Ôćĺ Exchanging (self-generates `Arrived` in Arriving entry action)
- Elevator: Exchanging Ôćĺ Idle via `Door_closed` (guard: next_stop_floor == current_floor)

Requires:
- Action-assignment direction fix (#1)
- `Up`/`Down` enum scoping fix (#2)
- `action_Arriving_entry` compiled (or stubbed to just generate `Arrived`)
- `action_Exchanging_entry` compiled (or stubbed to just send `Open_cmd` to CarDoor)

### Scenario 3: Button-Triggered Travel
FloorCallButton at floor 3 receives `Activated`. Its entry action navigates
to elevator and sends `Floor_assigned`. Elevator travels same path as #2.

Requires: all of #2 plus `action_Lit_entry` compiled (#3).

### Scenario 4: Full End-to-End
Button pressed Ôćĺ elevator travels to floor Ôćĺ arrives Ôćĺ door opens and closes
Ôćĺ elevator goes Idle.

Combines scenario 3 (travel) + scenario 1 (door cycle). Requires all of #2
and #3 plus `action_Exchanging_entry` compiled (#5) to send `Open_cmd` to
CarDoor.

### Scenario 5: Passenger Round-Trip
1. FloorCallButton at floor 1 receives `Activated` ÔÇö elevator wakes from Idle
2. Elevator travels from floor 1 to floor 1 (already there) or from another floor
3. Door opens (via Exchanging entry action Ôćĺ CarDoor)
4. User presses DestFloorButton for floor 3 (`Activated` Ôćĺ DestFloorButton Lit)
5. Door closes (CarDoor Close_cmd after 5s delay Ôćĺ Closed Ôćĺ `Door_closed` to Elevator)
6. Elevator guard: `next_stop_floor != current_floor` Ôćĺ Exchanging Ôćĺ Departing
7. Elevator travels to floor 3
8. Arrives Ôćĺ Exchanging entry action notifies DestFloorButton (Call_served Ôćĺ Lit Ôćĺ Idle), opens door
9. Door opens and closes
10. Elevator guard: `next_stop_floor == current_floor` Ôćĺ Exchanging Ôćĺ Idle

Requires: everything from #4 plus `action_Arriving_entry` compiled (#4) to
pick next stop, and DestFloorButton state machine working.
