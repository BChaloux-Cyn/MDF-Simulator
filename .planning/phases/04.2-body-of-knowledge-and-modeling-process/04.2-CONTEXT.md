# Phase 04.2: Body of Knowledge and Modeling Process - Context

**Gathered:** 2026-03-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Produce BoK (Body of Knowledge) documentation that enables a fresh Claude context — with no prior history of this project — to create a valid, well-formed MDF domain model. Content is derived from patterns observed during Phase 4.1 action body authoring. Also produce a draft subagent strategy for model development (gather → plan → execute, similar to GSD). Implementation work (schema, model, tests) belongs in Phase 4.1, not here.

</domain>

<decisions>
## Implementation Decisions

### Location

- BoK documents live in `docs/` for now
- Will eventually move to `mdf-server` repo (these are agent-facing, not library-facing)
- No special naming convention required — Claude's discretion on file names and structure

### Target audience

A fresh Claude context with:
- No memory of this project's history
- Access to `docs/` BoK documents and `schema/` + `pycca/` reference files
- A modeling task (e.g., "model a traffic light controller")
- Should be able to produce valid YAML that passes `validate_model` without asking the user clarifying questions about schema syntax

### Content scope

Documents should cover at minimum:
- **Domain modeling guide**: what a domain is, when to split into multiple domains, domain bridge patterns
- **Class modeling guide**: entity vs. active stereotypes, identifier selection, attribute types, referential attributes and when to add them, subtype/supertype patterns
- **State machine authoring guide**: how to structure states, transitions, events, guards, entry/exit actions using pycca
- **Association patterns**: 1:1, 1:*, *:*, self-referential (linked list), subtype formalization
- **Common modeling pitfalls**: patterns observed as errors or near-misses during Phase 4.1 authoring

### Subagent strategy (draft — informed by Phase 4.1 experience)

The strategy for how to subdivide domain modeling labor across subagents is **not defined in advance**. Phase 4.1 provides the experience base. Questions to answer in Phase 4.2:
- Is there a gather → plan → execute pattern analogous to GSD?
- Can class diagram authoring be separated from state machine authoring?
- What context does a subagent need to author a single class in isolation?
- What coordination is required between subagents (referential integrity, event naming)?

The Phase 4.2 planning agent should read the Phase 4.1 action body authoring notes (logged in the BoK as patterns) and derive the strategy from that experience, not from first principles.

### Claude's Discretion

- Exact document names and structure within `docs/`
- How many documents vs. one comprehensive guide
- Level of detail in each section
- How to structure the subagent strategy document (if it exists as a separate doc)

</decisions>
