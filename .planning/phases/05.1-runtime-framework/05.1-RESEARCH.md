## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Instance Registry
- D-01: Registry manages create, delete, and lookup by class name + identifier attribute values
- D-02: Composite identifiers supported (multi-attribute keys)
- D-03: Only concrete subtype instances are instantiated; supertypes are modeling constructs only
- D-04: Supertype attribute inheritance — subtypes carry all supertype attributes, accessible via `self.attr`
- D-05: Sync creation places instance directly in specified state (no entry action). Async creation posts creation event to standard queue (entry action executes normally).
- D-06: Sync deletion removes instance immediately (no final action). Async deletion posts deletion event to standard queue (final action executes before removal).

#### Relationship Link Store
- D-07: Stores links between instance pairs keyed by relationship ID (e.g., "R5")
- D-08: Enforces multiplicity from schema (1:1, 1:M, M:M) — violation produces an error micro-step
- D-09: Supports `relate`, `unrelate`, and `navigate` (including chained traversal like `self->R1->R2->R3`)

#### Three-Queue Event Scheduler
- D-10: Priority queue — self-directed events (sender == receiver instance). FIFO.
- D-11: Standard queue — cross-instance events, creation events, deletion events. FIFO.
- D-12: Delay queue — pending delayed events with remaining-time counters; feeds standard queue on expiry. Never dispatched directly, never to priority.
- D-13: Scheduler always checks priority queue before standard queue. After each event completes, re-check priority before next standard dequeue.
- D-14: At most one delayed signal of a given event type per sender/receiver pair pending at any time. Posting a second cancels the first.
- D-15: Scheduler halts when all three queues are empty.
- D-16: FIFO ordering within each queue — required for determinism.
- D-17: Two runs of the same scenario must produce identical micro-step streams.

#### Run-to-Completion
- D-18: One event processes fully — all micro-steps including any `generate` calls enqueued — before the scheduler selects the next event.
- D-19: A generated event during action execution is enqueued but not dispatched until the current event completes.

#### Event Dispatch
- D-20: Polymorphic event dispatch — events directed at a supertype are routed to the subtype's state machine.
- D-21: "Can't happen" produces an error micro-step. "Event ignored" silently consumes the event (no micro-step beyond `event_received`).
- D-22: Event parameters carried as args dict on event objects. Action code accesses via `rcvd_evt["param"]`.

#### Final State and Deletion
- D-23: Final state detection — states with no outgoing transitions are final states.
- D-24: Reaching a final state triggers automatic async deletion (posts deletion event to standard queue).

#### Cancel Semantics
- D-25: `cancel Event from sender to target` removes a pending delayed event matching on event type + sender instance + target instance in the delay queue.

#### Micro-Step Stream
- D-26: All 12 micro-step types: `scheduler_selected`, `event_received`, `guard_evaluated`, `transition_fired`, `action_executed`, `generate_dispatched`, `event_delayed`, `event_delay_expired`, `event_cancelled`, `instance_created`, `instance_deleted`, `bridge_called`.
- D-27: Every side effect yields a discrete micro-step from the generator. Both CLI and GUI consume this stream.
- D-28: Error conditions yield error micro-steps — never Python exceptions. Covers: unknown event target, unparseable action, double-deletion, can't-happen event, multiplicity violation.

#### Bridge Mock Registry
- D-29: YAML file mapping `{operation: return_value}`. Loaded at startup.
- D-30: Bridge calls hit the registry and record `bridge_called` micro-step. Undefined operations return None without error.

#### Simulation Clock
- D-31: Simulation clock (not wall clock) drives the delay queue.
- D-32: Speed multiplier affects only delay queue expiry — not action execution speed.
- D-33: Clock supports pause and resume (consumed by debugger in 5.4/5.5).
- D-34: `now()` returns current simulation clock time.

#### ctx Runtime Object
- D-35: `ctx` is the single API surface that generated action code calls into. Owns: scheduler, instance registry, relationship store, bridge mock registry, clock.
- D-36: Methods: `generate()`, `cancel()`, `create_sync()`, `create_async()`, `delete()`, `relate()`, `unrelate()`, `navigate()`, `select_any()`, `select_many()`, `bridge()`, `now()`.

#### Self-Contained Package
- D-37: `engine/` must have zero imports from `schema/`, `tools/`, or `pycca/`. It stands alone when copied into the opaque bundle.

## Summary

Phase 5.1 builds the pure Python runtime framework that generated action code executes against. It is a self-contained package under `engine/` with zero imports from `schema/`, `tools/`, or `pycca/`. The framework implements all 22 execution domain rules from the design doc, exposes a generator-based micro-step stream, and provides a `ctx` runtime object as the single API surface.

The behavioral specification is fully locked in `docs/2026-03-06-simulation-engine-design.md` (22 numbered rules) and the 37 decisions in `05.1-CONTEXT.md`. No external library research is needed for the core framework — it is pure Python using stdlib data structures. The only open design choices (Claude's discretion) are module layout and internal data structure selection, both of which have clear standard answers for this problem domain.

The exit gate for this phase is a hand-written test harness that exercises all 12 micro-step types and all scheduler rules without requiring the compiler (5.2). This means the framework must accept a domain manifest supplied as plain Python dicts, not compiled bundles.

**Primary recommendation:** Implement `engine/` as a multi-module package (`registry.py`, `scheduler.py`, `relationship.py`, `clock.py`, `ctx.py`, `microstep.py`) with a thin `__init__.py` re-exporting the public surface. Use `dataclasses` for micro-steps and events. Use `collections.deque` for priority and standard queues. Use a sorted list (or `heapq`) for the delay queue keyed by expiry time.

### Pattern 1: Generator-Based Micro-Step Stream

The simulation engine is a Python generator function. Every side effect yields one `MicroStep`. Consumers (`for step in run_simulation(...)`) never need to call into the engine imperatively.

```python
# Source: docs/2026-03-06-simulation-engine-design.md + D-27
from typing import Generator

def run_simulation(
    domain_manifest: dict,
    scenario: dict,
    bridge_mocks: dict | None = None,
) -> Generator["MicroStep", None, None]:
    ctx = SimulationContext(domain_manifest, bridge_mocks)
    ctx.load_scenario(scenario)
    yield from ctx.execute()
```

The generator contract is the protocol between `engine/` and all consumers (CLI in 5.4, GUI in 5.5, MCP tool wrappers in 5.3). Changing the generator protocol is a breaking change for all downstream phases.

### Pattern 2: Run-to-Completion via In-Progress Buffer

During event processing, `generate()` calls enqueue into a local "pending" buffer, not the scheduler. Only when the current event processing completes does the buffer flush to the scheduler. This enforces D-18/D-19 without needing a flag.

```python
# Source: design doc rules 10-11 + D-18/D-19 [ASSUMED for implementation shape]
class ThreeQueueScheduler:
    def process_event(self, event: Event) -> Generator[MicroStep, None, None]:
        self._in_flight = event
        self._pending_generated: list[Event] = []   # run-to-completion buffer
        yield from self._dispatch(event)
        # Flush pending to queues only after dispatch completes
        for generated in self._pending_generated:
            self._enqueue(generated)
        self._in_flight = None
```

### Pattern 3: Dataclass Micro-Steps with Type Discriminator

All 12 micro-step types are dataclasses sharing a common base. The `type` field is a string literal that consumers can switch on without isinstance chains.

```python
# Source: design doc micro-step table + D-26 [ASSUMED for representation]
from dataclasses import dataclass, field
from typing import Literal

@dataclass
class MicroStep:
    """Base — never instantiated directly."""
    type: str

@dataclass
class SchedulerSelected(MicroStep):
    type: Literal["scheduler_selected"] = field(default="scheduler_selected", init=False)
    queue: Literal["priority", "standard"]
    event_type: str
    target_class: str
    target_instance_id: dict   # composite key dict

@dataclass
class EventReceived(MicroStep):
    type: Literal["event_received"] = field(default="event_received", init=False)
    class_name: str
    instance_id: dict
    event_type: str
    args: dict
    queue: Literal["priority", "standard"]

# ... (all 12 types follow this pattern)
```

### Pattern 4: Instance Identity as Frozen Dict Key

Instance identity is a composite of one or more identifier attribute values. Store as a `frozenset` of `(attr_name, value)` pairs for hashability as dict keys. Expose to consumers as a plain `dict` in micro-step payloads.

```python
# Source: D-01, D-02, design doc §Instance Identity [ASSUMED for implementation]
def make_instance_key(identifier_attrs: dict) -> frozenset:
    """Convert {attr: value} identifier dict to hashable registry key."""
    return frozenset(identifier_attrs.items())
```

### Pattern 5: Delay Queue with Simulation Clock

The delay queue entries are `(expiry_time, event)` tuples in a min-heap. On each scheduler tick, entries with `expiry_time <= clock.now()` are moved to the standard queue. The heap provides O(log n) expiry checks.

```python
# Source: design doc rules 4, 8 + D-12, D-31 [ASSUMED for implementation]
import heapq

class ThreeQueueScheduler:
    def __init__(self):
        self._priority: deque[Event] = deque()
        self._standard: deque[Event] = deque()
        self._delay: list[tuple[float, Event]] = []  # min-heap by expiry

    def tick_delay_queue(self, clock: "SimulationClock") -> list[MicroStep]:
        steps = []
        now = clock.now()
        while self._delay and self._delay[0][0] <= now:
            _, event = heapq.heappop(self._delay)
            self._standard.append(event)
            steps.append(EventDelayExpired(...))
        return steps
```

Pass multiplicity metadata into the relationship store at initialization from the domain manifest. Enforce at `relate()` time, not at schema-load time.

### Anti-Patterns to Avoid

- **Using wall-clock time in the delay queue:** The scheduler must use simulation clock time (D-31). Wall-clock calls (`time.time()`) anywhere in the engine are a bug.
- **Raising Python exceptions for "can't happen" or multiplicity violations:** Error conditions must yield error micro-steps (D-28), not raise. The consumer decides whether to halt.
- **Importing from `schema/`, `tools/`, or `pycca/`:** Strict isolation is a non-negotiable design requirement (D-37). Any such import causes the opaque bundle to break.
- **Using a global registry or scheduler instance:** The `ctx` object owns all components. Multiple simultaneous simulations must be possible without interference.
- **Non-FIFO priority or standard queue dispatch:** Any ordered structure other than a strict deque violates the determinism requirement (D-16, D-17).
- **Dispatching generated events immediately during action execution:** Generated events must buffer until the current event completes (D-18/D-19). Direct dispatch skips run-to-completion.

## Common Pitfalls

### Pitfall 1: Supertype Instance Creation
**What goes wrong:** Creating a supertype instance in the registry because the caller passed the supertype class name (e.g., "Call") rather than the concrete subtype ("ElevatorCall").
**Why it happens:** D-03 prohibits supertype instantiation; D-20 requires polymorphic dispatch through the subtype's state machine. If the registry stores a "Call" instance, dispatch fails.
**How to avoid:** The domain manifest must identify each class as concrete or abstract (supertype). Registry `create` must validate and reject abstract class names with an error micro-step.
**Warning signs:** A test that creates a supertype instance and then dispatches an event to it — if it doesn't produce an error micro-step, the guard is missing.

### Pitfall 2: Delay Queue Determinism Under Equal Expiry Times
**What goes wrong:** Two events with the same expiry time are moved to standard queue in non-deterministic order (heap tie-breaking by Python object identity).
**Why it happens:** `heapq` compares tuples left-to-right; if `expiry_time` values are equal, it falls through to comparing the `Event` object, which may not implement `__lt__`.
**How to avoid:** Add a monotonic sequence counter as the second element of the heap tuple: `(expiry_time, sequence_number, event)`. Sequence number enforces FIFO within same-expiry events.
**Warning signs:** Test D-17 (two identical runs) occasionally fails under concurrent or randomized event generation.

### Pitfall 3: Double-Dispatch of Generated Events
**What goes wrong:** An event generated inside an action fires immediately (priority queue checked mid-action) rather than after the current event fully completes.
**Why it happens:** If the scheduler loop is re-entered recursively or polled inside `_dispatch()`, run-to-completion is violated.
**How to avoid:** Scheduler must not process queue while `_in_flight` is set. Generated events go to a pending buffer; buffer only flushes after `_dispatch()` returns.
**Warning signs:** Test sequence where `generate X to self` during action A causes X to fire before A's remaining micro-steps complete.

### Pitfall 4: Relationship Navigate Returns Wrong Type
**What goes wrong:** `navigate(self, "R2")` returns a list when multiplicity is `0..1` or `1`, or returns a single instance when multiplicity is `0..*`.
**Why it happens:** The relationship store must know the multiplicity to decide return type. If multiplicity data is missing from the domain manifest, it defaults incorrectly.
**How to avoid:** Domain manifest is the authority on multiplicity; the manifest is required to specify both sides of every association. `navigate()` raises `KeyError` (not silently wrong type) when relationship metadata is missing.
**Warning signs:** A `select any related by self->R1` succeeds but returns a list of length 1, breaking downstream `.value()` calls.

### Pitfall 5: YAML Boolean Coercion in Bridge Mocks
**What goes wrong:** Bridge mock YAML contains `return_value: true` (YAML boolean), but action code expects Python `True`. Or `return_value: Off` is coerced to `False` by PyYAML 1.1.
**Why it happens:** PyYAML 1.1 treats `On`, `Off`, `Yes`, `No`, `True`, `False` as booleans. This is a known project pitfall recorded in STATE.md (Phase 04.1 decision: "YAML boolean keywords must be quoted").
**How to avoid:** Load bridge mock YAML with `yaml.safe_load()`. Document that `On`/`Off`/`Yes`/`No` as string return values must be quoted in the mock YAML.
**Warning signs:** `bridge("IsTopFloor", ...)` returns `False` when mock file says `return_value: No`.

### Pitfall 6: Final State Detection Relies on Schema Data
**What goes wrong:** The framework cannot detect final states (D-23) because it has zero imports from `schema/`. If final-state detection requires reading `StateDef.terminal`, the engine would need the schema.
**Why it happens:** Schema isolation (D-37) conflicts with needing class metadata.
**How to avoid:** The domain manifest (passed from outside, produced by the compiler in 5.2, or supplied by the test harness by hand) must include a `final_states` list per class. The engine reads this from the manifest, not from schema. For the Phase 5.1 hand-written test harness, this is supplied directly in the test fixture.
**Warning signs:** Final state detection never fires in any test — or fires unconditionally on all states.

### Pitfall 7: ctx Methods Called Before Engine is Running
**What goes wrong:** `ctx.generate()` called at module import time or in a fixture setup before the simulation generator is started.
**Why it happens:** The generator is lazy — it doesn't run until `next()` is called. Pre-flight ctx calls bypass the generator entirely.
**How to avoid:** `ctx` methods that generate micro-steps should assert that the generator is active (`self._running` flag). All instance setup goes through explicit `ctx.create_sync()` calls inside the generator after it starts.
**Warning signs:** `bridge_called` micro-steps appear in the test but the generator hasn't been iterated.

### Three-Queue Scheduler Loop
```python
# Source: design doc scheduler rules 6-9 + D-13, D-15 [ASSUMED]
def execute(self) -> Generator[MicroStep, None, None]:
    while True:
        self._tick_delay_queue()   # move expired entries to standard
        if self._priority:
            event = self._priority.popleft()
            yield SchedulerSelected(queue="priority", ...)
            yield from self._process_event(event)
        elif self._standard:
            event = self._standard.popleft()
            yield SchedulerSelected(queue="standard", ...)
            yield from self._process_event(event)
        else:
            break  # all queues empty — domain idle (rule 9)
```

### Instance Registry Lookup
```python
# Source: D-01, D-02 [ASSUMED]
class InstanceRegistry:
    def __init__(self):
        # {class_name: {frozenset(identifier_items): instance_dict}}
        self._store: dict[str, dict[frozenset, dict]] = {}

    def lookup(self, class_name: str, identifier: dict) -> dict | None:
        key = frozenset(identifier.items())
        return self._store.get(class_name, {}).get(key)
```

### Cancel Semantics
```python
# Source: D-14, D-25 [ASSUMED]
def cancel(self, event_type: str, sender_id: dict, target_id: dict) -> list[MicroStep]:
    """Remove matching pending delayed event from delay queue."""
    match_key = (event_type, frozenset(sender_id.items()), frozenset(target_id.items()))
    before_len = len(self._delay)
    self._delay = [
        (exp, seq, e) for (exp, seq, e) in self._delay
        if (e.event_type, frozenset(e.sender_id.items()), frozenset(e.target_id.items())) != match_key
    ]
    heapq.heapify(self._delay)
    if len(self._delay) < before_len:
        return [EventCancelled(cancelled_event_type=event_type, ...)]
    return []
```

## Open Questions

1. **Domain manifest schema**
   - What we know: The compiler (5.2) produces the manifest; the framework consumes it. The test harness in 5.1 hand-writes it.
   - What's unclear: Exact dict structure of the domain manifest — field names for class list, association list, event catalog, final state flags, generalization hierarchy.
   - Recommendation: Define the manifest structure in `engine/manifest.py` as a plain dict spec (with a docstring) during Wave 1. The compiler in 5.2 must produce conforming dicts.

2. **Transition table representation**
   - What we know: Generated classes (from 5.2) hold transition tables and action functions. In 5.1, the test harness supplies these directly.
   - What's unclear: How does the test harness supply a transition table without compiled code? It must be a dict mapping `{(current_state, event_type): (next_state, action_fn | None, guard_fn | None)}`.
   - Recommendation: Define `TransitionTable = dict[tuple[str, str], TransitionEntry]` in the framework and document it as the interface the compiler must satisfy in 5.2.

3. **Polymorphic dispatch routing when sender targets a supertype**
   - What we know: D-20 says events directed at a supertype are routed to the subtype's state machine.
   - What's unclear: At dispatch time, does the scheduler receive the event with the supertype class name and resolve to subtype, or does `ctx.generate()` validate and substitute immediately?
   - Recommendation: `ctx.generate()` should resolve the concrete instance (subtype) at enqueue time, so the scheduler always sees concrete class names. This avoids supertype lookups at dispatch time.

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Malformed bridge mock YAML (DoS via deeply nested structures) | Denial of Service | Use `yaml.safe_load()` not `yaml.load()` — already project convention per PyYAML docs [VERIFIED: STATE.md established pyyaml usage] |
| Unbounded delay queue growth (infinite loops via delayed generates) | Denial of Service | The scheduler halts when all queues are empty (D-15). Delay-queue growth from recurring events is bounded by the model — no framework-level mitigation needed for v1. |
| Infinite recursion via self-events without final state | Denial of Service | Out of scope for v1 — the framework does not implement a cycle detector. Document as known limitation. |

**Note:** This is a local simulation library with no network exposure. Security risk surface is limited to YAML input parsing.
