# Phase 5.1: Runtime Framework - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Build the runtime framework that generated code executes against: instance registry, relationship link store, three-queue event scheduler, `ctx` runtime object, micro-step stream generator, bridge mock registry, and simulation clock. The framework must be self-contained (no imports into `schema/`, `tools/`, or `pycca/`) so it can be copied into opaque bundles.

Exit gate: Framework runs a hand-written test harness exercising all runtime components — no compiler needed yet.

</domain>

<decisions>
## Implementation Decisions

### Instance registry
- **D-01:** Registry manages create, delete, and lookup of class instances by class name + identifier attribute values
- **D-02:** Composite identifiers supported (multi-attribute keys)
- **D-03:** Only concrete subtype instances are instantiated — supertypes are modeling constructs only
- **D-04:** Supertype attribute inheritance — subtypes carry all supertype attributes, accessible via `self.attr`
- **D-05:** Sync creation places instance directly in specified state (no entry action). Async creation posts creation event to standard queue (entry action executes normally).
- **D-06:** Sync deletion removes instance immediately (no final action). Async deletion posts deletion event to standard queue (final action executes before removal).

### Relationship link store
- **D-07:** Stores links between instance pairs keyed by relationship ID (e.g., "R5")
- **D-08:** Enforces multiplicity from schema (1:1, 1:M, M:M) — violation produces an error micro-step
- **D-09:** Supports `relate`, `unrelate`, and `navigate` (including chained traversal like `self->R1->R2->R3`)

### Three-queue event scheduler
- **D-10:** Priority queue — self-directed events (sender == receiver instance). FIFO.
- **D-11:** Standard queue — cross-instance events, creation events, deletion events. FIFO.
- **D-12:** Delay queue — pending delayed events with remaining-time counters; feeds standard queue on expiry. Never dispatched directly, never to priority.
- **D-13:** Scheduler always checks priority queue before standard queue. After each event completes, re-check priority before next standard dequeue.
- **D-14:** At most one delayed signal of a given event type per sender/receiver pair pending at any time. Posting a second cancels the first.
- **D-15:** Scheduler halts when all three queues are empty.
- **D-16:** FIFO ordering within each queue — required for determinism.
- **D-17:** Two runs of the same scenario must produce identical micro-step streams.

### Run-to-completion
- **D-18:** One event processes fully — all micro-steps including any `generate` calls enqueued — before the scheduler selects the next event.
- **D-19:** A generated event during action execution is enqueued but not dispatched until the current event completes.

### Event dispatch
- **D-20:** Polymorphic event dispatch — events directed at a supertype are routed to the subtype's state machine.
- **D-21:** "Can't happen" vs "event ignored" — per state/event cell in the transition table. "Can't happen" produces an error micro-step. "Event ignored" silently consumes the event (no micro-step beyond `event_received`).
- **D-22:** Event parameters carried as args dict on event objects. Action code accesses via `rcvd_evt["param"]`.

### Final state and deletion
- **D-23:** Final state detection — states with no outgoing transitions are final states.
- **D-24:** Reaching a final state triggers automatic async deletion (posts deletion event to standard queue per design doc rule 18).

### Cancel semantics
- **D-25:** `cancel Event from sender to target` removes a pending delayed event matching on event type + sender instance + target instance in the delay queue.

### Micro-step stream
- **D-26:** All 12 micro-step types from the design doc: `scheduler_selected`, `event_received`, `guard_evaluated`, `transition_fired`, `action_executed`, `generate_dispatched`, `event_delayed`, `event_delay_expired`, `event_cancelled`, `instance_created`, `instance_deleted`, `bridge_called`.
- **D-27:** Every side effect yields a discrete micro-step from the generator. Both CLI and GUI consume this stream.
- **D-28:** Error conditions yield error micro-steps — never Python exceptions. Covers: unknown event target, unparseable action, double-deletion, can't-happen event, multiplicity violation.

### Bridge mock registry
- **D-29:** YAML file mapping `{operation: return_value}`. Loaded at startup.
- **D-30:** Bridge calls hit the registry and record `bridge_called` micro-step. Undefined operations return null without error.

### Simulation clock
- **D-31:** Simulation clock (not wall clock) drives the delay queue.
- **D-32:** Speed multiplier affects only delay queue expiry — not action execution speed.
- **D-33:** Clock supports pause and resume (consumed by debugger in 5.4/5.5).
- **D-34:** `now()` returns current simulation clock time.

### ctx runtime object
- **D-35:** `ctx` is the single API surface that generated action code calls into. Owns: scheduler, instance registry, relationship store, bridge mock registry, clock.
- **D-36:** Methods: `generate()`, `cancel()`, `create_sync()`, `create_async()`, `delete()`, `relate()`, `unrelate()`, `navigate()`, `select_any()`, `select_many()`, `bridge()`, `now()`.

### Self-contained package
- **D-37:** `engine/` must have zero imports from `schema/`, `tools/`, or `pycca/`. It stands alone when copied into the opaque bundle.

### Claude's Discretion
- Internal data structures for each component (deque vs list, dict vs custom class, etc.)
- Module split within `engine/` (single file vs multiple — e.g., scheduler.py, registry.py, etc.)
- Micro-step representation (dataclasses, NamedTuple, TypedDict — whatever is cleanest)
- Test harness structure for the exit gate

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `engine/__init__.py` — Currently a stub. Will become the runtime framework package.
- `schema/yaml_schema.py` — Defines Association multiplicity, ClassDef identifiers, generalization hierarchies. The compiler (5.2) reads these; the runtime enforces them. Runtime needs to know the shapes but not import the schema module — relationship metadata is passed in via the compiled bundle's domain manifest.

### Established Patterns
- Pydantic models use `populate_by_name=True` — runtime data structures should follow similar flexibility if loading from YAML mocks
- Flat uv layout (no `src/` directory) — `engine/` sits at package root

### Integration Points
- `engine/` is consumed by the compiler (5.2) which copies it into bundles
- `engine/` is consumed by the simulation runner (5.3) which loads bundles and executes
- `engine/` is consumed by MCP tool wrappers in `tools/simulation.py` (5.3)
- The micro-step stream protocol is the contract between engine and all consumers (CLI, GUI, MCP tools)

</code_context>

## Specific Ideas

- The 22 execution domain rules in the design doc are the behavioral spec — they must all be satisfied
- The micro-step stream is the universal output contract — every consumer (CLI, GUI, MCP) reads the same stream
- Determinism is non-negotiable — same input must produce identical output across runs
