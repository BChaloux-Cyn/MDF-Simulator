# Phase 05.1 Plan 04: Three-Queue Scheduler Summary

Built `ThreeQueueScheduler` — the core execution loop of the simulation engine. Implements priority/standard/delay queue routing, generator-based dispatch, run-to-completion semantics, polymorphic supertype dispatch, can't-happen vs event-ignored handling, final-state async deletion, and cancel-with-at-most-one-per-triple delay queue invariants. All 22 execution-domain rules from the design doc that touch dispatch converge here.

## What Shipped

- `engine/scheduler.py` — `ThreeQueueScheduler`
  - `enqueue(event)` routes to priority/standard/delay per D-10..D-14, or to `_pending_generated` when an event is in flight (D-18, D-19)
  - `cancel(event_type, sender_class, sender_id, target_class, target_id)` removes a matching entry from the delay queue and returns `[EventCancelled(...)]` per D-25
  - `tick_delay_queue()` pops all expired entries and re-enqueues them onto the standard queue, yielding `EventDelayExpired` micro-steps (rule 8)
  - `execute()` is a generator that yields micro-steps until all queues are empty (rule 9). Always ticks the delay queue first, then drains priority before standard (rules 6-7).
  - `_process_event()` handles target resolution (with polymorphic supertype walk), `EventReceived`, transition table lookup, can't-happen vs event-ignored vs guard vs transition execution, and final-state async deletion
  - Special handling for `__creation__` (sets initial state, runs optional entry action) and `__deletion__` (delegates to `registry.process_deletion`)

## Deviations from Plan

**1. [Rule 2 - Critical functionality] action_fn signature gained scheduler arg**

- **Found during:** Implementing `test_run_to_completion`
- **Issue:** The plan said action_fn "may call back into the scheduler's `enqueue()` via a callback passed as parameter" but didn't pin the signature. Without passing the scheduler in, generated events from inside an action have nowhere to go.
- **Fix:** action_fn is now called as `action_fn(instance, args, scheduler)` with a `TypeError` fallback to `action_fn(instance, args)` so existing 2-arg `_noop_action` lambdas still work.
- **Files modified:** `engine/scheduler.py`
- **Commit:** rolled into `a624c83`
