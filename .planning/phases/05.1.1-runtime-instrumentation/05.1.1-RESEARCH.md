## Summary

Phase 5.1.1 is a pure-additive instrumentation pass on top of the completed Phase 5.1 runtime framework. The implementation touches exactly five source files: four `engine/` modules plus the test file. No schema, no compiler, no CLI changes.

The central challenge is identifying the precise insertion points in `ThreeQueueScheduler._process_event()` and `_process_creation()` for the four new micro-step types. Timing wraps the entire event-dispatch cycle; senescence checks attach to two specific moments within it.

A notable discrepancy exists between the CONTEXT.md's naming (`StateEntered`, `TransitionTaken`) and the actual micro-step names in the codebase (`TransitionFired`). There is no `StateEntered` micro-step in the engine — CONTEXT.md uses that name as a conceptual label for the moment a state transition completes. The plan must map CONTEXT decision language to actual code locations.

**Primary recommendation:** Implement timing first (simplest path to `EventCompleted`), then add the senescence sidecar dict and emit logic, then wire `LongEventWarning`. Each can be verified independently against the D-22 test criteria.

### Pattern 3: Timing Insertion Points in `_process_event`

The current `_process_event` flow [VERIFIED: engine/scheduler.py lines 296-436]:

```
1. Special cases: __creation__ / __deletion__ delegated
2. _resolve_target → concrete_class, concrete_id
3. registry.get_state
4. yield EventReceived           ← existing
5. lookup transition table entry
6. guard evaluation → yield GuardEvaluated
7. self._in_flight = event
8. action_fn call → yield ActionExecuted
9. registry.set_state → yield TransitionFired
10. final state check → buffer delete event
11. finally: flush _pending_generated, self._in_flight = None
```

**Timing start** goes between step 1 and step 2 (after popping from queue, before yielding `EventReceived`). The pop happens in `execute()` — so timing starts in `execute()` before calling `_process_event`, or at the very top of `_process_event` before the first yield. [ASSUMED — CONTEXT D-02 says "after the scheduler pops the event from a queue and just before yielding EventDispatched"; in the actual code the equivalent is `EventReceived`, not `EventDispatched` — see Mismatch #1 below]

**Timing end + EventCompleted** goes after the `finally` block in `_process_event` completes and before the scheduler's main loop resumes for the next event. The cleanest placement is back in `execute()` after `_process_event` returns, still within the same loop iteration.

**`LongEventWarning`** is emitted in `execute()` immediately after yielding `EventCompleted`, if `ctx.event_duration_warn_ns` is set and `duration_ns > threshold`. The scheduler does not currently hold a reference to `SimulationContext` — this attribute must be accessible. [See Mismatch #2 below]

### Anti-Patterns to Avoid

- **Folding `LongEventWarning` into `EventCompleted` as a field:** breaks the separate-micro-step contract (D-09); trace consumers must be able to filter on type
- **Using `Required` / removing `NotRequired` for `senescent_states`:** breaks backward compat with the 42 existing tests whose manifests omit the field
- **Emitting `SenescentEntered` for final states:** D-15 explicitly prohibits this; check final-state deletion before emitting
- **Putting timing calls inside `_process_event`:** `EventCompleted` should bracket the entire dispatch including RTC flush; timing should live in `execute()`
- **Importing `SimulationContext` from `ctx.py` in `scheduler.py`:** circular import; `event_duration_warn_ns` must be passed to the scheduler at construction or accessed via a protocol/callback [see Mismatch #2]

## Critical Code/Context Mismatches

These are discrepancies between the CONTEXT.md decision language and the actual engine code, confirmed by direct code reading. The plan MUST resolve each one.

### Mismatch #1: "EventDispatched" vs "EventReceived" / "StateEntered" vs "TransitionFired"

**CONTEXT says:** D-02 "just before yielding `EventDispatched`"; D-19 canonical sequence uses `TransitionTaken(from, to, event)` and `StateEntered(instance, new_state)`.

**Code has:** No `EventDispatched` micro-step and no `StateEntered` micro-step. The scheduler yields `EventReceived` (line 339) when an event is dispatched to an instance, and `TransitionFired` (line 416) when a state transition occurs. These are the Phase 5.1 micro-step names (verified in `microstep.py`).

**Resolution for the plan:**
- "EventDispatched" in D-02 = `EventReceived` in the code. Timing start is immediately before `yield EventReceived`.
- "TransitionTaken" in D-19 = `TransitionFired` in the code.
- "StateEntered" in D-13/D-18/D-19 = the moment `registry.set_state()` is called and `TransitionFired` is yielded; senescence check runs after `yield TransitionFired`.
- For `_process_creation`, "StateEntered" = after `registry.set_state(target_class, target_id, initial_state)` is called (line 454 of scheduler.py).

[VERIFIED: engine/microstep.py — 12 types enumerated, no EventDispatched or StateEntered]
[VERIFIED: engine/scheduler.py — EventReceived at line 339, TransitionFired at line 416]

### Mismatch #2: `ctx.event_duration_warn_ns` accessibility from scheduler

**CONTEXT says:** D-05 puts `event_duration_warn_ns` on `SimulationContext` (`ctx.py`). D-20 says the scheduler's main loop emits `LongEventWarning`.

**Code has:** `ThreeQueueScheduler` does not hold a reference to `SimulationContext`. The scheduler is constructed inside `SimulationContext.__init__` and only passed `registry`, `clock`, `class_defs`, `generalizations`. `ctx.py` calls `self.scheduler.execute()` and yields from it — the scheduler is completely decoupled from the ctx.

**Resolution options for the plan (choose one):**
1. **Pass threshold as a callable/reference at construction:** add `warn_ns_fn: Callable[[], int | None] | None = None` parameter to `ThreeQueueScheduler.__init__`; ctx passes `lambda: self.event_duration_warn_ns`
2. **Store threshold directly on scheduler:** add `event_duration_warn_ns: int | None = None` to the scheduler (not ctx) and expose it as a property that ctx proxies; `ctx.event_duration_warn_ns` becomes a property that reads/writes `self.scheduler.event_duration_warn_ns`
3. **Emit `LongEventWarning` in ctx.execute():** the ctx wraps `scheduler.execute()` in its own generator; it inspects each yielded `EventCompleted` step and appends `LongEventWarning` when threshold exceeded — keeps scheduler free of ctx knowledge

[ASSUMED — this is the most important discretion decision; option 3 is cleanest because it keeps `ThreeQueueScheduler` free of ctx coupling and preserves the established pattern]

### Mismatch #3: `_process_event` doesn't return for event-ignored case

**Current code (line 370):**
```python
if next_state is None and action_fn is None and guard_fn is None:
    return
```
This early return happens before timing end. `EventCompleted` must still be yielded even for event-ignored dispatches if timing started — otherwise the consumer sees a timing-start with no matching `EventCompleted`. [ASSUMED — correct interpretation; the plan must handle this case]

### Mismatch #4: Error-path early returns in `_process_event`

Lines 314–326 (`unknown_target`), 330–335 (`unknown_class`), 363 (guard false), 381-391 (`guard_error`) — all return early. Timing started before any of these; `EventCompleted` must be yielded for all paths including errors, or timing must not start until after the target-resolution check. [ASSUMED — simplest fix is to start timing after target resolution succeeds; trade-off is that target-resolution time is not captured, which is acceptable]

## Common Pitfalls

### Pitfall 1: `SenescentEntered` emitted for final states

**What goes wrong:** After `TransitionFired` into a final state, the senescence check runs and `SenescentEntered` fires. Then `InstanceDeleted` fires. This is wrong per D-15.

**Why it happens:** Both final-state check and senescence check run after `TransitionFired`. Order of the two `if` blocks matters.

**How to avoid:** Final-state check runs first. Only if the instance is NOT going to be deleted does the senescence check run. In code: check `final_states` before checking `senescent_states`.

**Warning signs:** `test_final_state_async_deletion` (existing) plus new D-22 test #7 will both fail if order is wrong.

### Pitfall 2: Double-emit of `SenescentEntered` on senescent→senescent self-transition

**What goes wrong:** An instance in a senescent state receives an event that transitions it to another senescent state. The sidecar flag is still True. If the flag is not consulted, `SenescentEntered` fires twice.

**Why it happens:** D-16 requires the flag to prevent this, but the flag logic must also handle the case where `SenescentExited` cleared the flag at the start of dispatch and the new state is also senescent.

**How to avoid:** Per D-16, `SenescentExited` fires when dispatch starts (flag cleared), so after the transition the flag is False — `SenescentEntered` fires once correctly. No special case needed; the standard flag-check handles it.

### Pitfall 3: `time.perf_counter_ns()` start placed inside RTC buffer flush

**What goes wrong:** If timing start is placed inside the `try` block that handles the RTC buffer, the flush time of generated events is not counted — but D-02 says end point is "after run-to-completion drains all generated events."

**Why it happens:** `_pending_generated` is flushed in the `finally` block by calling `_enqueue_direct` for each buffered event. The timing end must be after this flush, not inside the try block.

**How to avoid:** Start timing in `execute()` before calling `_process_event()`; stop timing in `execute()` after `_process_event()` returns. The entire dispatch + flush is bracketed.

### Pitfall 4: `NotRequired` import missing

**What goes wrong:** `from typing import NotRequired` is absent; the `ClassManifest` TypedDict uses `senescent_states: set[str]` without `NotRequired`, making it a required field — existing tests fail with `TypedError` on missing key.

**Why it happens:** `typing.NotRequired` was added in Python 3.11. It's available (project requires >=3.11) but not currently imported in `manifest.py`.

**How to avoid:** Add `NotRequired` to the import from `typing` at the top of `manifest.py`.

### Pitfall 5: `ctx.py` imports `ThreeQueueScheduler` — circular import risk

**What goes wrong:** If `scheduler.py` is modified to import from `ctx.py` (to access `event_duration_warn_ns`), a circular import occurs because `ctx.py` already imports `scheduler.py`.

**Why it happens:** `ctx.py` line 17: `from engine.scheduler import ThreeQueueScheduler`.

**How to avoid:** Choose Option 3 from Mismatch #2 above — handle `LongEventWarning` in `ctx.execute()`, not in the scheduler. Alternatively pass a callable. Never have `scheduler.py` import `ctx.py`.

## Open Questions

1. **`EventCompleted.target` and `.name` field format**
   - What we know: CONTEXT uses `target: str` and `name: str` without specifying what `target` is (class name? instance key? both?)
   - What's unclear: Is `target` the class name, the instance key, or a compound string like `"Controller:1"`?
   - Recommendation: Match the format used by `EventReceived` — `class_name: str` and `instance_id: dict` — but CONTEXT explicitly says `str`. Use `f"{concrete_class}:{make_instance_key(concrete_id)}"` for consistency with how `SenescentEntered.instance` will be formatted.

2. **`LongEventWarning` threshold check location (Mismatch #2)**
   - What we know: CONTEXT puts the attribute on `SimulationContext`; scheduler has no ctx reference
   - What's unclear: Which of the three options the user prefers
   - Recommendation: Option 3 (ctx.execute() wrapper) because it requires zero changes to the scheduler constructor signature and zero new coupling. The plan should document this as a discretion decision made by the implementer.

3. **`SenescentExited` when `SenescentEntered` was never fired**
   - What we know: D-14 says `SenescentExited` fires when the dispatched instance's current state is in `senescent_states`. The flag is in `_senescent`.
   - What's unclear: What if an instance was created into a senescent state (D-18 fires `SenescentEntered`), then the flag is set, then `SenescentExited` fires when woken — this is the happy path. But what if a manifest is hand-loaded with `senescent_states` and the instance is already in a senescent state from a previous simulation step without the flag being set? (Can only happen if senescence dict is not initialized on first `set_state`.)
   - Recommendation: Initialize the flag to False for all new instances on creation; never check state before first event.
