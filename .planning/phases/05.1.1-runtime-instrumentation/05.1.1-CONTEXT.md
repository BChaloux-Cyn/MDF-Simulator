# Phase 5.1.1: Runtime Instrumentation - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Add runtime observability micro-steps to the Phase 5.1 engine **without modifying any existing behavior**:

- `EventCompleted` micro-step with `duration_ns` measured per event dispatch
- Opt-in soft warning threshold (`ctx.event_duration_warn_ns`) emitting `LongEventWarning` when exceeded
- `SenescentEntered` / `SenescentExited` micro-steps marking instance idleness
- `senescent_states: set[str]` field on `ClassManifest` so the scheduler can detect senescence

This is **pure additive runtime work** — no compiler involvement, no schema changes, no breaking changes to the existing 42 engine tests. Phase 5.2 (Model Compiler) will populate `senescent_states` from action body analysis; this phase only adds the runtime field and the scheduler logic that consumes it.
</domain>

<decisions>
## Implementation Decisions

### Event execution timing

- **D-01:** Add a new immutable dataclass `EventCompleted(target: str, name: str, duration_ns: int)` to `engine/microstep.py`. Yielded once per event dispatch, after all transition / action / state-entry micro-steps but **before** any `SenescentEntered` for the same dispatch.
- **D-02:** Timing measured via `time.perf_counter_ns()`. Start point: immediately after the scheduler pops the event from a queue and just before yielding `EventDispatched`. End point: immediately after run-to-completion drains all generated events for the in-flight dispatch and the scheduler returns to its main loop for the next event.
- **D-03:** Timing instrumentation runs on **every** event dispatch — no opt-out, no global flag. Cost is two `perf_counter_ns()` calls per event (~100 ns each on modern hardware). Negligible compared to action body execution.
- **D-04:** Wall-clock timing only. Virtual simulation time (from `SimulationClock`) is unaffected and independent — `duration_ns` measures real elapsed CPU/wall time, not the model's notion of time.

### Soft warning threshold

- **D-05:** Add `event_duration_warn_ns: int | None` attribute to `SimulationContext` (`engine/ctx.py`), default `None`. When `None`, no `LongEventWarning` is ever emitted. When set to a positive integer, every `EventCompleted` whose `duration_ns` exceeds the threshold is followed by a `LongEventWarning(target, name, duration_ns, threshold_ns)` micro-step.
- **D-06:** Add immutable dataclass `LongEventWarning(target: str, name: str, duration_ns: int, threshold_ns: int)` to `engine/microstep.py`.
- **D-07:** Threshold is **opt-in** (per user preference). No default value, no auto-detection. Modeler sets it explicitly when they want the safety net: `ctx.event_duration_warn_ns = 100_000_000` for 100 ms.
- **D-08:** Threshold can be changed mid-simulation by writing the attribute. The next `EventCompleted` uses the new value. No state mutation needed beyond the attribute write.
- **D-09:** `LongEventWarning` is emitted as a separate micro-step (not folded into `EventCompleted`) so consumers can filter on the warning type cleanly without inspecting every `EventCompleted`.

### Senescent state detection

- **D-10:** Add `senescent_states: set[str]` field to `ClassManifest` TypedDict in `engine/manifest.py`. Default value: `set()` (empty). Backward compatible — existing 42 tests pass because they don't reference the field, and an empty set means no instance is ever classified as senescent.
- **D-11:** Add immutable dataclass `SenescentEntered(instance: str, state: str, settled_at: int)` to `engine/microstep.py`. `settled_at` is virtual time from `SimulationClock.now()` at the moment of senescence entry — useful for dwell-time computation in Phase 5.6 metrics.
- **D-12:** Add immutable dataclass `SenescentExited(instance: str, state: str, exited_at: int, by_event: str)` to `engine/microstep.py`. `exited_at` is virtual time at exit. `by_event` names the event that woke the instance up.
- **D-13:** **Emission rule for `SenescentEntered`:** After `StateEntered(instance, state)` is yielded, the scheduler checks `manifest.class_defs[class_of(instance)].senescent_states`. If `state` is in that set, yield `SenescentEntered(instance, state, ctx.clock.now())` immediately after `StateEntered`. The check is O(1) (set membership).
- **D-14:** **Emission rule for `SenescentExited`:** When the scheduler pops an event from a queue and the target instance's *current* state is in its class's `senescent_states`, yield `SenescentExited(instance, state, ctx.clock.now(), event_name)` **before** `EventDispatched`. The trace order is: `... [time passes] → SenescentExited → EventDispatched → GuardEvaluated → ...`.
- **D-15:** **Final state wins over senescence.** If an event's processing transitions the instance into a final state (which triggers async deletion per Phase 5.1 SC-07), `SenescentEntered` is **not** emitted for that final state, even if the final state happens to be in `senescent_states`. Final-state detection runs first; senescence detection only runs if the instance is still alive after RTC.
- **D-16:** **No double-emission of `SenescentEntered`.** The scheduler maintains a per-instance "currently senescent" flag. `SenescentEntered` only fires on the *transition into* senescence — i.e., when the flag was False and is now True. If an event dispatches to a senescent instance and the new state is also senescent (rare but legal: senescent → senescent transition), no new `SenescentEntered` is emitted; the instance was already senescent and remains so. `SenescentExited` is still emitted when the dispatch begins because the *previous* state was senescent.
- **D-17:** **Senescence flag storage.** The flag lives in a sidecar `dict[str, bool]` owned by `ThreeQueueScheduler`, keyed by instance key. This avoids polluting `InstanceRegistry`'s instance dicts (which are shared with generated code via `self`). On `InstanceDeleted`, the entry is removed.
- **D-18:** **Initial state senescence.** When a newly-created instance is placed into its initial state via `InstanceCreated → StateEntered`, the senescence check runs normally. If the initial state is senescent, `SenescentEntered` fires immediately after `StateEntered` — the instance was created, settled, and is waiting.

### Micro-step ordering for the full event-dispatch sequence

- **D-19:** Canonical order for one event dispatch on an instance that starts senescent and ends senescent in a different state:
  ```
  SenescentExited(instance, prev_state, exited_at, event_name)   # only if prev_state was senescent
  EventDispatched(target, name, params)
  GuardEvaluated(...)              # zero or more
  TransitionTaken(from, to, event)
  ActionExecuted(name)             # zero or more, may yield: AttributeWritten, EventGenerated, BridgeCalled, Related, Unrelated, InstanceCreated, InstanceDeleted
  StateEntered(instance, new_state)
  ActionExecuted(name)             # entry action of new_state, if any (with the same nested micro-steps as above)
  EventCompleted(target, name, duration_ns)
  LongEventWarning(...)            # only if threshold set and duration_ns > threshold_ns
  SenescentEntered(instance, new_state, settled_at)   # only if new_state is senescent and prev was not (or it's a genuine re-entry on a non-senescent → senescent path)
  ```
- **D-20:** Order is enforced by the scheduler's main loop, not by individual micro-step emitters. New helper methods on the scheduler (`_emit_event_completed`, `_emit_senescent_check`, `_emit_senescent_exit_if_needed`) wrap the existing dispatch flow.

### Backward compatibility

- **D-21:** All 42 existing tests in `tests/test_engine.py` must pass without modification. If any test breaks because the trace gained a new micro-step it didn't expect, the test was over-specified — fix the test, not the engine. *Acceptable change to existing tests:* relaxing assertions that previously checked exact trace length or trace equality. *Unacceptable:* changing the meaning of any pre-5.1.1 micro-step type.
- **D-22:** New tests cover every new code path:
  1. `EventCompleted` is emitted exactly once per event dispatch with a positive `duration_ns`
  2. `LongEventWarning` is **not** emitted when threshold is `None` (default)
  3. `LongEventWarning` is emitted when a long-running action exceeds the threshold (test uses `time.sleep` in a fake action body)
  4. `SenescentEntered` fires on transition into a senescent state from a non-senescent state
  5. `SenescentExited` fires on dispatch out of a senescent state, naming the waking event
  6. **No double-emit:** `SenescentEntered` fires once even across senescent → senescent self-transitions
  7. **Final state wins:** an instance entering a state that is both senescent AND final emits `InstanceDeleted` (existing) and does NOT emit `SenescentEntered`
  8. Initial state senescence: a newly-created instance whose initial state is senescent emits `SenescentEntered` immediately after `StateEntered`
  9. `senescent_states` field defaults to `set()` and the scheduler never emits senescence micro-steps for classes that don't populate it
  10. Manifest with multiple classes, mixed senescent / non-senescent: each class's classification is independent

### Phase boundaries

- **D-23:** This phase **does not** classify states as senescent or transient. It only **consumes** the `senescent_states` field. Classification is Phase 5.2 compiler work (per `05.2-CONTEXT.md` D-14). For Phase 5.1.1 tests, manifests are hand-built with `senescent_states` populated explicitly.
- **D-24:** This phase **does not** add anything to `engine/ctx.py` beyond the `event_duration_warn_ns` attribute. No new public API methods, no new configuration system.
- **D-25:** This phase **does not** modify `engine/registry.py`, `engine/relationship.py`, `engine/clock.py`, or `engine/bridge.py`. All changes are confined to `engine/microstep.py`, `engine/manifest.py`, `engine/scheduler.py`, `engine/ctx.py`, `engine/__init__.py` (re-exports), and `tests/test_engine.py`.

### Claude's Discretion

- Helper method names inside `ThreeQueueScheduler` (e.g., `_emit_senescent_check` vs `_check_senescence_and_emit`) — pick what reads naturally
- Internal storage type for the senescence flag dict (built-in `dict` vs `defaultdict` vs `WeakValueDictionary`) — built-in `dict` is fine; weakness isn't needed because we explicitly remove on `InstanceDeleted`
- Test fixture organization — extend the existing `tests/test_engine.py` or split into a new `tests/test_engine_instrumentation.py` based on whether the file feels too large
- Whether to add a `__repr__` override on the new dataclasses — frozen dataclasses already get a reasonable default
</decisions>

## Existing Code Insights

### Reusable Assets

- **Micro-step dataclass pattern** (`engine/microstep.py`): all 12 existing types use `@dataclass(frozen=True)`. New types follow the same pattern. Re-exported via `engine/__init__.py`.
- **Scheduler dispatch loop** (`engine/scheduler.py`): the `run_simulation` generator and the internal `dispatch_one` / `_in_flight` machinery is where every existing micro-step is yielded. New emissions plug in at the natural boundaries: before `EventDispatched` (for `SenescentExited`), and after the RTC-completed point where the next iteration of the main loop would start (for `EventCompleted`, `LongEventWarning`, `SenescentEntered`).
- **`SimulationClock.now()`** (`engine/clock.py`): returns virtual time. Already used by the delay queue. New senescence micro-steps use it for their `settled_at` / `exited_at` fields.
- **`InstanceRegistry.lookup_class()`** (`engine/registry.py`): given an instance key, returns its class name. Used by the scheduler to look up `senescent_states` for the correct class manifest. (If this method doesn't exist by exactly that name, the scheduler already has equivalent access — to be confirmed at plan time.)

### Established Patterns

- **Frozen dataclasses for all micro-steps** — immutability is load-bearing; consumers of the trace rely on it. New types must follow.
- **Per-class manifest fields are TypedDict keys** — `ClassManifest` is a `TypedDict`, so adding `senescent_states` is a TypedDict expansion. Hand-built manifests in tests can omit it (default applied at lookup time via `cls_manifest.get("senescent_states", set())`).
- **Test fixtures use hand-built manifests** — Phase 5.1 tests construct `DomainManifest` literals directly rather than going through any compiler. Phase 5.1.1 tests do the same; populating `senescent_states` is just `{"Idle"}` in a literal.
- **Tests assert on the micro-step stream, not internal state** — every behavioral assertion is "given input X, the trace contains [list of micro-steps]". New tests follow the same pattern.
- **No imports outside `engine/`** — Phase 5.1 SC-11. `engine/` package has zero imports from `schema/`, `tools/`, `pycca/`. Phase 5.1.1 must not break this.

### Integration Points

- **`engine/microstep.py`** — Add 4 new dataclasses (`EventCompleted`, `LongEventWarning`, `SenescentEntered`, `SenescentExited`).
- **`engine/manifest.py`** — Add `senescent_states: set[str]` field to `ClassManifest`. Use `NotRequired` from `typing` to keep it optional for backward compat with hand-built manifests in existing tests.
- **`engine/scheduler.py`** — Add timing measurement around dispatch, senescence flag tracking dict, helper methods for emitting the four new micro-step types at the right moments. The bulk of the new code lives here.
- **`engine/ctx.py`** — Add `event_duration_warn_ns: int | None = None` attribute. One line plus docstring.
- **`engine/__init__.py`** — Re-export the four new micro-step types alongside the existing 12.
- **`tests/test_engine.py`** — Append new test functions for each acceptance criterion. Don't modify existing tests unless they over-specify trace length (they shouldn't — Phase 5.1 tests check for *containment* of expected micro-steps, not exact equality).
</code_context>

<specifics>
## Specific Ideas

- The user's debugging scenario: "If a while loop in an action body runs away, I want to see it in the trace immediately." This drives the timing instrumentation design — `EventCompleted` brackets every event so any pathological duration is one micro-step away from visibility.
- Senescent state semantics come from `docs/MODEL-BUILDING-PROCESS.md`: a senescent state is a *modeled resting state*, not a runtime accident. The user wants the trace to clearly mark "this instance has done its work and is parked." `SenescentEntered`/`SenescentExited` provide unambiguous brackets.
- The user specifically said the warning threshold should be **opt-in** (no default). Rationale: avoids noise on cold-start CI machines, doesn't punish modelers who haven't tuned their thresholds yet.
- `LongEventWarning` is a separate micro-step (not a flag on `EventCompleted`) so trace consumers can grep for warnings without parsing every duration field.
</specifics>

<deferred>
## Deferred Ideas

- **Senescence dwell-time histograms** — aggregating how long instances spent in each senescent state. This is a Phase 5.6 (Metrics Gatherer) concern; 5.1.1 only emits the raw micro-steps. The metrics gatherer consumes them and builds histograms.
- **Flow-aware senescence classification** — compiler analyzing conditional branches in entry actions to determine "this state is sometimes transient, sometimes senescent." Out of scope for both 5.1.1 (runtime) and 5.2 (compiler uses purely syntactic rule per 05.2 D-14).
- **`SenescenceBroken` micro-step** — a third senescence-related type that fires when an external event arrives at a senescent instance. Considered and rejected: `SenescentExited` already serves this purpose with the `by_event` field. Adding a fourth type would duplicate information.
- **Per-class or per-state thresholds** — `event_duration_warn_ns` is a single global value. Per-class overrides could be added in a future phase if needed.
- **Histogram of `duration_ns` accumulated by the engine** — only the raw `EventCompleted` is emitted; aggregation belongs in Phase 5.6.
- **Hard kill on long events** — only the soft warning is added. A hard timeout that aborts a runaway event would require interrupting Python execution mid-action, which is non-trivial and out of scope.
</deferred>
