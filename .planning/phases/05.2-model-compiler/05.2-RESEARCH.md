## Summary

Phase 5.2 builds a new `compiler/` package that reads YAML model files (via `schema/`), parses action bodies (via `pycca/`), and emits Python source files per class plus a JSON manifest, packaged into a `.mdfbundle` zip. Three independent workstreams exist: (1) grammar fixes needed before codegen can proceed, (2) the Lark Transformer codegen core, and (3) bundle packaging and determinism.

The most critical finding is a **mismatch between D-13's stated encoding for can't_happen/event_ignored and the encoding already implemented in `engine/scheduler.py`**. The scheduler tests confirm the current encoding is: absent key = cant_happen (ErrorMicroStep), present with all-None = event_ignored. D-13 describes the inverse. The planner must pick one encoding and reconcile both the compiler output and the scheduler before Phase 5.3 loader work begins.

The second critical finding is **four grammar gaps** in `pycca/grammar.py` that will cause parse failures on real elevator model actions. These must be fixed in Wave 0 or Wave 1 before the transformer can be wired up.

**Primary recommendation:** Fix grammar gaps first (Wave 0), then build transformer + manifest builder, then bundle packager, then elevator integration test. Plan as 5 sequential plans.

### Pattern 1: Lark Transformer for Code Generation

**What:** Subclass `lark.Transformer`. Each grammar rule method receives its children already transformed (bottom-up). Return a Python source string from each method.

**When to use:** All action body and guard expression codegen.

**Example (verified pattern):**
```python
# Source: lark 1.3.1 docs + verified via .venv test
from lark import Transformer, Token, Tree, v_args

class ActionTransformer(Transformer):
    # Bottom-up: children are already strings when a rule method is called

    def name(self, children):
        return str(children[0])

    def dotted_name(self, children):
        # self.attr → inst["attr"] (translate self access to dict lookup)
        obj, attr = str(children[0]), str(children[1])
        if obj == "self":
            return f'self_dict["{attr}"]'
        return f'{obj}["{attr}"]'

    def compare_expr(self, children):
        if len(children) == 3:
            return f"{children[0]} {children[1]} {children[2]}"
        return children[0]

    def assignment(self, children):
        # self.attr = expr; → self_dict["attr"] = expr
        attr, expr = str(children[0]), children[1]
        return f'self_dict["{attr}"] = {expr}'

    def start(self, children):
        return "\n".join(children)
```

**Transformer traversal order:** Bottom-up (leaf → root). Each rule method fires after all its children are transformed. Methods for ambiguous `?expr` inline rules are not called — use the concrete rule names (`or_expr`, `and_expr`, etc.).

**Key insight:** The grammar uses `?expr` which collapses transparent rules. Always handle `compare_expr`, `add_expr`, `mul_expr`, `or_expr`, `and_expr` explicitly.

### Pattern 2: Source Location via `propagate_positions=True`

**What:** Pass `propagate_positions=True` to `Lark(...)` — every `Tree` node gets a `.meta` attribute with `.line` and `.column`.

**When to use:** For `# from <model_file>:<line>` comments on every emitted block (D-05).

**Example:**
```python
# Source: verified via .venv test
from lark import Lark
p = Lark(PYCCA_GRAMMAR, start="start", parser="earley", propagate_positions=True)
tree = p.parse(action_body)
# tree.children[0].meta.line, tree.children[0].meta.column give YAML-relative positions
```

**Caveat:** `meta.line` is relative to the parsed string, not the YAML file. The compiler must add the YAML block's starting line number to get absolute source locations. [VERIFIED: .venv test]

### Pattern 4: Bundle Packaging for Determinism

```python
# Source: D-07, D-12 (from CONTEXT.md); zipfile stdlib verified
import zipfile, json, hashlib
from pathlib import Path

def write_bundle(domain_name: str, files: dict[str, str], manifest: dict,
                 engine_version: str, pycca_version: str) -> Path:
    # Sort all dicts by key before serialization (D-07)
    manifest_json = json.dumps(manifest, sort_keys=True, indent=2)
    content_hash = hashlib.sha256(
        json.dumps(sorted(files.items()), ensure_ascii=True).encode()
    ).hexdigest()
    bundle_meta = {
        "engine_version": engine_version,
        "pycca_version": pycca_version,
        "model_hash": content_hash,
        # NO timestamp field — D-07
    }
    bundle_path = Path(f"{domain_name}.mdfbundle")
    with zipfile.ZipFile(bundle_path, "w", compression=zipfile.ZIP_DEFLATED) as zf:
        # Write in sorted order for determinism
        zf.writestr("bundle.json", json.dumps(bundle_meta, sort_keys=True))
        zf.writestr("manifest.json", manifest_json)
        zf.writestr("generated/__init__.py", "")
        for class_name in sorted(files.keys()):
            zf.writestr(f"generated/{class_name}.py", files[class_name])
    return bundle_path
```

### Anti-Patterns to Avoid

- **Sorting at write-time only:** Python dict ordering is insertion-order stable but `set` iteration is not. Any `set` in the model (e.g., `senescent_states`) must be converted with `sorted()` before serialization.
- **Using `ast.Module` for codegen:** Correct approach is string templates (D-01 rejection rationale).
- **Calling `exec()` on generated code in-process:** D-01 explicitly rejects this.
- **Importing from `engine/` in `compiler/`:** D-11 hard constraint. Use `engine.manifest` TypedDicts for type hints only (use `TYPE_CHECKING` guard).
- **Passing `ctx` to guard functions:** The scheduler calls `guard_fn(self_dict, params_dict)` with **two arguments**, not three. D-10 describes a 3-arg signature but the scheduler implementation uses 2-arg with a `try/except TypeError` fallback. Generated guard functions should use the 2-arg signature that the scheduler actually invokes.

## Grammar Gap Analysis (CRITICAL)

**All 4 gaps verified by running elevator model actions against `pycca/grammar.py`** [VERIFIED: .venv tests].

### Gap 1: Lambda return type cannot be generic

**Rule affected:** `lambda_expr`
**Current:** `lambda_expr: "[" capture_list? "]" PIPE lambda_params PIPE "->" NAME "{" ...`
**Problem:** `-> Set<FloorCallButton>` fails because grammar expects `NAME` not `generic_type`.
**Elevator impact:** `Elevator.Arriving` entry action (`flat_map` lambda), `Elevator._pick_next_stop` method.
**Fix:** Change `-> NAME` to `-> type_expr` in `lambda_expr` rule.

### Gap 2: Capture list cannot contain attribute access

**Rule affected:** `capture_list`
**Current:** `capture_list: NAME ("," NAME)*`
**Problem:** `[self.current_floor]` and `[my_floor]` fail when `my_floor` is a dotted name from the outer scope or when `self.attr` is captured.
**Elevator impact:** `Elevator.Arriving` filter lambda, `Elevator.Exchanging` select-where lambda, `CarDoor.Opening` filter lambda.
**Fix:** Change `capture_list: NAME ("," NAME)*` to `capture_list: capture_item ("," capture_item)*` where `capture_item: NAME "." NAME | NAME`.

**Note:** Capture list items of the form `self.attr` are unusual — the model likely intends to capture the *value* of `self.current_floor` into the lambda's closure. The compiler translates `[self.current_floor]` to a captured local variable binding in the generated Python closure.

### Gap 3: `select_related_stmt` only handles single-hop traversal

**Rule affected:** `select_related_stmt`
**Current:** `"select" ("any"|"many") NAME "related" "by" NAME "->" NAME ";"`
**Problem:** `select many stops related by self->R1->R2;` fails at second `->`.
**Elevator impact:** Would affect any statement-form select with chained traversal. The `select_expr` (RHS of typed_var_decl) already supports `traversal_chain` with multiple hops — the statement form needs to match.
**Fix:** Change to `"select" ("any"|"many") NAME "related" "by" traversal_chain ";"` (reuse existing `traversal_chain` rule).

### Gap 4: Bridge call uses named params (`k: v`) but grammar expects positional `arglist`

**Rule affected:** `bridge_call`
**Current:** `NAME "::" NAME "[" arglist? "]" ";"`
**Problem:** `Building::IsTopFloor[floor_num: self.current_floor]` fails because `arglist` expects `expr` not `NAME ":" expr`.
**Elevator impact:** Any bridge call that passes named arguments (the `Transport::ElevatorDetected` bridge body uses `arg.sensor_id` — a different issue; see Gap 2). The Building bridge calls in `validate_model` tests use named params.
**Fix:** Add `named_arg_list: NAME ":" expr ("," NAME ":" expr)*` rule and update `bridge_call` to accept either `arglist` or `named_arg_list`.

## Critical Finding: can't_happen / event_ignored Encoding Mismatch

**D-13 states:**
- `event_ignored` → cell absent (scheduler treats missing key as silent ignore)
- `can't_happen` → cell present with `next_state=None, action_fn=None, guard_fn=None` AND sentinel

**`engine/scheduler.py` (lines 381-404) implements:**
- `entry is None` (absent key) → `ErrorMicroStep(error_kind="cant_happen")` — i.e., **absent = cant_happen**
- `entry present with all-None` → silent return — i.e., **present-all-None = event_ignored**

**Tests confirm the scheduler encoding** (see `test_cant_happen_error_microstep` and `test_event_ignored_silent` in `tests/test_engine.py`).

**Impact:** The compiler must match whatever encoding the scheduler expects, not D-13's stated intent. D-13 was written before the scheduler was tested. The planner must:
1. Accept the scheduler's encoding as the ground truth (absent = cant_happen, present-all-None = event_ignored), OR
2. Update the scheduler to match D-13's encoding and update the existing tests

**Recommendation:** Accept scheduler encoding as ground truth (it has tests). The compiler emits cells for event_ignored transitions (present-all-None) and omits cells for can't_happen (absent). Add a regression test that cross-checks the compiler output against the scheduler's `test_cant_happen_error_microstep` test.

## Critical Finding: guard_fn Actual Call Signature

**D-10 specifies:** `def guard_name(ctx: SimulationContext, self: dict, params: dict) -> bool` (3-arg)

**`engine/scheduler.py` line 410-413 calls:**
```python
result = bool(guard_fn(
    self._registry.lookup(concrete_class, concrete_id),
    dict(event.args),
))
```
That is a 2-arg call: `guard_fn(self_dict, params_dict)`. There is no `try/except TypeError` fallback for guard functions (only for action functions). [VERIFIED: grep of scheduler.py]

**Impact:** Generated guard functions must accept exactly `(self_dict: dict, params: dict) -> bool` unless the scheduler is updated to pass `ctx` to guards. The planner must reconcile D-10 with the scheduler implementation.

## Common Pitfalls

### Pitfall 1: Transformer Bottom-Up Order vs. Statement Order

**What goes wrong:** Multi-statement action body arrives at `start` rule as a list of already-transformed children. If statements are joined with `"\n"` but some statements return `None` (e.g., unimplemented rules), silent `None` appears in output.
**Why it happens:** `Transformer.__default__` returns a `Tree` for unhandled rules, not a string. Template assembly then fails on mixed Tree/string lists.
**How to avoid:** Implement every grammar rule that appears in real action bodies, even if minimally. Add a `__default__` that raises `NotImplementedError` with the rule name so unhandled rules fail loudly.
**Warning signs:** Generated `.py` file contains `None` or `Tree(...)` as a literal.

### Pitfall 2: Determinism Broken by Set Iteration

**What goes wrong:** `ClassManifest.senescent_states` is a `set[str]`. When serialized to `manifest.json`, set-to-list conversion varies by insertion order.
**Why it happens:** Python sets have non-deterministic iteration order across processes.
**How to avoid:** Always `sorted(manifest["senescent_states"])` before JSON serialization. Apply the same discipline to all dict values that are sets.
**Warning signs:** Two consecutive builds produce different `manifest.json` content for `senescent_states`.

### Pitfall 3: black Not Installed

**What goes wrong:** `import black` raises `ModuleNotFoundError` at compile time.
**Why it happens:** `black` is not in `requirements.in`. D-01 requires it but it is not yet a project dependency.
**How to avoid:** Wave 0 task: add `black>=26.0` to `requirements.in` and run `uv pip install black`. [VERIFIED: black 26.3.1 available via uv pip dry-run]
**Warning signs:** `ModuleNotFoundError: No module named 'black'` when running `compile_model`.

### Pitfall 4: `propagate_positions` Adds Per-Parse Overhead

**What goes wrong:** Creating a second Lark parser instance with `propagate_positions=True` for source location tracking adds parse time. Using `STATEMENT_PARSER` from `grammar.py` (no positions) is faster but loses location info.
**Why it happens:** `STATEMENT_PARSER` in `grammar.py` was compiled without `propagate_positions`.
**How to avoid:** Compiler creates its own `Lark(PYCCA_GRAMMAR, ..., propagate_positions=True)` instances. The `STATEMENT_PARSER` and `GUARD_PARSER` exports remain unchanged (validators use them without position tracking).
**Warning signs:** `tree.meta.line` returns `None` or `AttributeError`.

### Pitfall 5: `self` in Grammar is Just NAME Token

**What goes wrong:** The grammar treats `self` as a plain `NAME` token, not a keyword. The transformer must recognise `self` in `dotted_name` and `name` rules and emit `self_dict[...]` instead of a variable access.
**Why it happens:** Grammar keeps `self` as `NAME` for simplicity (it's not a C/Python keyword in pycca).
**How to avoid:** In transformer's `dotted_name` rule: `if obj == "self": return f'self_dict["{attr}"]'`. In `name` rule: `if n == "self": return "self_dict"`.
**Warning signs:** Generated code contains `self["attr"]` instead of `self_dict["attr"]`, or bare `self` as variable name (illegal in generated Python).

### Pitfall 6: `rcvd_evt` Must Map to `params`

**What goes wrong:** `rcvd_evt.floor_num` in pycca is event parameter access. Generator must emit `params["floor_num"]`, not `rcvd_evt["floor_num"]`.
**Why it happens:** Grammar treats `rcvd_evt` as a plain `NAME` token.
**How to avoid:** In transformer's `dotted_name` rule: `if obj == "rcvd_evt": return f'params["{attr}"]'`.
**Warning signs:** Generated code has `rcvd_evt["floor_num"]` which will raise `NameError` at runtime.

### Pitfall 7: Zip Write Order Must Be Deterministic

**What goes wrong:** `zipfile.ZipFile` does not sort entries. Two builds with the same content but different OS directory listing order produce different zip byte sequences and thus different content hashes.
**Why it happens:** `os.listdir`, `Path.iterdir`, `dict.keys()` all have OS-dependent ordering when not explicitly sorted.
**How to avoid:** Always sort class names before writing: `for class_name in sorted(generated_files)`. Write `bundle.json` and `manifest.json` before `generated/` entries, always.
**Warning signs:** Determinism test (`test_two_builds_identical`) fails intermittently on different machines.

### Senescent State Classifier

```python
# Source: D-14 (CONTEXT.md) + lark Tree walk pattern [VERIFIED: .venv test]
from lark import Tree, Token

def _has_self_generate(tree: Tree | Token) -> bool:
    """Return True if tree contains any 'generate ... to self' statement.

    Conservative: any syntactic occurrence of generate-to-self classifies
    the state as transient, even if guarded. Per D-14 design rationale.
    """
    if isinstance(tree, Token):
        return False
    if tree.data == "generate_stmt":
        # generate_stmt: "generate" NAME "to" NAME ";"
        # or "generate" NAME "(" ... ")" "to" NAME ";"
        # The target NAME is always the last NAME child
        name_tokens = [c for c in tree.children if isinstance(c, Token) and c.type == "NAME"]
        if name_tokens and str(name_tokens[-1]) == "self":
            return True
    return any(_has_self_generate(child) for child in tree.children)

def classify_senescent_states(state_machine: StateDiagramFile,
                               parser: Lark) -> set[str]:
    """Return set of state names that are senescent (no generate-to-self in entry action)."""
    senescent = set()
    for state in state_machine.states:
        if state.entry_action is None:
            senescent.add(state.name)  # no action = no self-generate possible
            continue
        tree = parser.parse(state.entry_action)
        if not _has_self_generate(tree):
            senescent.add(state.name)
    return senescent
```

```python
def build_class_manifest(
    class_def: ClassDef,
    state_diagrams: dict[str, StateDiagramFile],  # class_name -> SD
    supertype: ClassDef | None,
    supertype_manifest: ClassManifest | None,
) -> ClassManifest:
    # --- Attribute flattening (D-04) ---
    attrs: dict[str, Any] = {}
    if supertype is not None:
        for a in supertype.attributes:
            attrs[a.name] = a.type  # supertype first
    for a in class_def.attributes:
        attrs[a.name] = a.type  # subtype wins on conflict

    # --- Transition table (D-03) ---
    own_sd = state_diagrams.get(class_def.name)
    super_sd = state_diagrams.get(supertype.name) if supertype else None

    if own_sd is not None:
        # Case A or C: build from own SM
        ttable = _build_transition_table(own_sd)
    elif supertype_manifest is not None and super_sd is not None:
        # Case B: copy supertype's table
        ttable = dict(supertype_manifest["transition_table"])
    else:
        ttable = {}  # entity class, no SM

    return ClassManifest(
        name=class_def.name,
        is_abstract=class_def.stereotype == "entity" and bool(class_def.partitions),
        identifier_attrs=_flatten_identifiers(class_def, supertype),
        attributes=dict(sorted(attrs.items())),  # D-07 sorted
        initial_state=own_sd.initial_state if own_sd else None,
        final_states=[s.name for s in (own_sd.states if own_sd else []) if s.terminal],
        transition_table=ttable,
        supertype=supertype.name if supertype else None,
        subtypes=[],  # filled in by manifest builder
        senescent_states=set(),  # filled in by senescence classifier
    )
```

   - What we know: `while`, `switch`, set `union`/`intersection`/`difference` are in SYNTAX.md §3 but not in `pycca/grammar.py`.

   - What we know: D-10 says 3-arg `(ctx, self, params)`; scheduler calls 2-arg `(self_dict, params)`.
   - What's unclear: Should the compiler emit 3-arg guards (and update the scheduler's guard call site), or emit 2-arg guards (matching scheduler today)?
   - Recommendation: Update the scheduler to pass `ctx` to guard functions (matching D-10), or confirm 2-arg is intentional. Either way, both the scheduler and generated code must agree.

   - What we know: Scheduler uses absent=cant_happen, present-all-None=event_ignored. D-13 says the inverse.
   - What's unclear: Is this a D-13 documentation error, or should the scheduler be changed?
   - Recommendation: Accept scheduler's encoding (it has green tests). Update D-13 comment in CONTEXT.md to match reality. Add a comment in compiler code explaining the divergence.

4. **Callable stripping from `manifest.json`**
   - What we know: D-02 says manifest is serialized "without callables; loader rebinds them." The `DomainManifest` TypedDict uses `Callable` in `TransitionEntry`.
   - What's unclear: The serialization strategy — does the compiler serialize the manifest with `None` for all callables, then the loader replaces them? Or are the per-class `.py` files the source of truth and `manifest.json` only holds structural data?
   - Recommendation: Compiler serializes structural data only (no callables). `manifest.json` uses a parallel dict schema where `action_fn`/`guard_fn` are stored as string function names (e.g., `"action_Departing_entry"`). The Phase 5.3 loader `import`s the generated `.py` file and rebinds by name.
