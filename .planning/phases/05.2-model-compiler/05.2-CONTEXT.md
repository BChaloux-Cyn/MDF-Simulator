# Phase 5.2: Model Compiler — Context

## Phase Goal (from ROADMAP)

Lark Transformer compiles pycca action bodies and guards into Python source. Compiler generates one file per class (transition tables + action functions), a domain manifest, and packages everything into an opaque self-contained zip bundle.

**Depends on:** Phase 5.1 (Runtime Framework)
**Acceptance:** Elevator model compiles to a bundle without errors; bundle is deterministic.

## Locked Decisions

### D-01: Codegen strategy — string templates + black
The Lark `Transformer` returns Python source as strings. Per-construct f-string templates assemble class files. Final pass formats every generated file through `black` for stable, readable output.

**Why:** Simple, debuggable, generated `.py` files are first-class artifacts that the GDB debugger phases (5.4/5.5) will inspect. The `ast` module adds weight without value for our line-oriented action language. Direct in-memory `exec` was rejected because invisible code is hostile to debugging.

### D-02: Bundle contents — generated code + manifest + pinned framework version
The opaque bundle is a zip containing:
- `generated/<class>.py` — one file per concrete class
- `manifest.json` — serialized `DomainManifest` (without callables; loader rebinds them)
- `bundle.json` — metadata: model content hash, pycca version, **`engine_version`** (pinned, no timestamp)

The bundle does **not** vendor a copy of `engine/`. The Phase 5.3 loader checks `engine_version` against the installed `engine/` and refuses incompatible bundles.

**Why:** Smaller bundles, single source of truth for `engine/`, defers the vendoring decision until we actually have multiple deployment targets. Compatibility check protects against silent runtime drift.

### D-03: Generalization — full replacement on subtype state machine override
Per the three-case analysis:

| Case | Supertype | Subtype | Compiler emits |
|------|-----------|---------|----------------|
| A | Entity (no SM) | Active (has SM) | Subtype's `transition_table` built from its own SM. No polymorphic dispatch needed. |
| B | Active (has SM) | Active, no SM redefinition | Subtype's `ClassManifest` gets a **copy** of supertype's `transition_table` (same `action_fn` / `guard_fn` references). |
| C | Active (has SM) | Active, redefines its SM | Subtype's `transition_table` is built **only** from its own state machine YAML. Supertype's transitions are NOT inherited or merged. |

**Polymorphic dispatch:** Already handled by `engine/scheduler.py` (Plan 05.1-04, SC-05). The scheduler walks `generalizations` to route supertype-named events to subtype tables. Compiler's job is just to populate per-class tables correctly.

**Why:** Each class's `transition_table` exactly mirrors what the modeler wrote — no hidden merging, no shadowing semantics, no ambiguous error attribution. If a subtype's table can't_happen an inherited supertype event, that's a real modeling error and surfaces as `ErrorMicroStep` at runtime.

### D-04: Attributes — always inherited and flattened into subtype manifest
Subtype `ClassManifest.attributes` contains all supertype attributes plus subtype's own. On name conflict, subtype wins. `identifier_attrs` likewise flattened. Matches the existing elevator model pattern (`CallButton → DestFloorButton/FloorCallButton` inherits `button_id`).

**Why:** Self-describing per-class manifests; no runtime walking of `generalizations` to read an attribute.

### D-05: Code style — readable + source line comments + type hints + black-formatted
Every emitted block of generated code is preceded by a `# from <model_file>:<line>` comment pointing back to the source YAML or pycca expression. Type hints on all generated functions. `black` formats the final output.

**Why:** Critical for the GDB debugger phases (5.4/5.5). Generated code must be inspectable by humans staring at a stack trace.

### D-06: Type system — strict NewType + runtime validation
- **MDF enums** → `enum.Enum` subclass
- **typedefs** (e.g., `type FloorNum = int`) → `typing.NewType('FloorNum', int)` for static-checker safety
- **Refinement constraints** (range, regex) → runtime validators called on every assignment to constrained attributes
- **Container types** (set/list/map) → builtin `set` / `list` / `dict` (no validation wrapper; element types enforced at write sites only)
- **Instance handles** → string keys (matches `make_instance_key` from Phase 5.1)

**Why:** Catches more bugs during development and surfaces type violations as compile or runtime errors with clear locations. The boilerplate cost is acceptable because generated code is auto-written, not hand-edited.

### D-07: Determinism — compiler enforces fully
- Every dict in the manifest is sorted by key before serialization (transition tables, attributes, associations, generalizations).
- Every iteration in generated code that affects observable state uses `sorted()`.
- `bundle.json` contains **no timestamps** — only the model content hash and pinned version strings.
- Verified by hashing two consecutive elevator builds and asserting equality (compile-time test).

**Why:** SC-08 demands "same input → identical bundle." Determinism at the compiler boundary is the only place to guarantee it without burdening the runtime.

### D-08: Action language scope — Tier 3 (full SYNTAX.md surface)
Compile **everything** documented in `pycca/SYNTAX.md`:

- All 14 statements from SC-01 (assignment, generate, cancel, create, delete, relate, unrelate, select, navigate, bridge, if/else, for-each, lambda, method calls)
- **Tier 1 expressions:** arithmetic (`+ - * / %`), comparison (`== != < > <= >=`), logical (`and or not`), boolean literals
- **Tier 2 container methods:** all `List<T>` / `Set<T>` / `Optional<T>` methods (size, is_empty, has_value, value, contains, find, filter, push/pop, get, peek, insert, remove)
- **Tier 2 time primitives:** `now()`, `Timestamp`, `Duration`, duration arithmetic, comparison, delayed `generate`, `cancel`
- **Tier 3:** `while`, `switch`, set operations (`union`, `intersection`, `difference`), `map` / `sort` with lambdas
- All forms of `select` (§9.1–§9.8): `select many` / `select any`, with/without `where`, with chained traversal
- `now()` built-in, `duration_s()`, `in_ms()` etc.

**Out of scope (compile error with clear message):**
- Anything in SYNTAX.md §12 "Differences from Standard OAL" that's marked as not supported
- Any construct not yet in the Lark grammar (`pycca/grammar.py`) — will surface during research as a planning input

**Why:** Front-loads the action-language work so Phase 5.3+ never has to revisit it. Acceptance test (elevator model) exercises a substantial slice of this surface; remaining surface needs to be tested independently.

### D-09: Compile error reporting — collect all, exit with full report
The compiler walks the entire model, accumulates every error with `file:line` source location and a human-readable message, then exits non-zero with the full list. Same UX as `validate_model` from Phase 3.

**Why:** Single round-trip per fix cycle. Modeler sees the full picture before reaching for the keyboard.

### D-10: Function signatures (locked, no question needed)
- **Action functions:** `def <action_name>(ctx: SimulationContext, self_dict: dict, params: dict) -> None`
- **Guard functions:** `def <guard_name>(self_dict: dict, params: dict) -> bool` (2-arg, per scheduler ground truth — superseded the original 3-arg form documented at planning time)
- `self_dict` is the instance dict from `InstanceRegistry`; `params` is the event parameter dict; `ctx` is the runtime API surface from Phase 5.1 (`engine/ctx.py`).
- **Note:** Guard functions are 2-arg because `engine/scheduler.py` calls `guard_fn(self_dict, params)` with exactly two positional arguments. Action functions remain 3-arg. This override is recorded in `05.2-RESEARCH.md` "Open Questions (RESOLVED)" §2.

### D-11: Compiler location and entry point (locked, no question needed)
- New package: `compiler/` (sibling to `engine/`, `schema/`, `tools/`, `pycca/`)
- Public API: `compile_model(model_root: Path, output_dir: Path) -> Path` returns path to the built bundle
- `compiler/` may import from `schema/` (to load YAML) and `pycca/` (to parse action source)
- `compiler/` MUST NOT import from `engine/` — it produces manifests, doesn't consume them. (Generated code imports `engine/`; the compiler itself does not.)

### D-12: Bundle filename convention (locked, no question needed)
- Filename: `<domain_name>.mdfbundle` (zip-format under the hood, custom extension to make intent obvious)
- Internal layout:
  ```
  <domain>.mdfbundle/
    bundle.json          # metadata: model_hash, engine_version, pycca_version
    manifest.json        # DomainManifest serialized (callables stripped — loader rebinds)
    generated/
      __init__.py
      <ClassA>.py
      <ClassB>.py
      ...
  ```

### D-13: can't_happen / event_ignored encoding (locked, no question needed)
Already constrained by Phase 5.1's scheduler:
- **`event_ignored`** → cell absent from `transition_table` (scheduler treats missing key as silent ignore)
- **`can't_happen`** → cell present with `next_state=None`, `action_fn=None`, `guard_fn=None` AND a sentinel marker (TBD in plan-time — likely a special `TransitionEntry` subclass or a magic `next_state="<cant_happen>"` string). Scheduler emits `ErrorMicroStep` on hit.
- Plan-time task: pick the encoding and add a regression test that round-trips through the scheduler's existing can't_happen test from Plan 05.1-04.

### D-14: Senescent state classification (compiler-side; runtime support comes from Phase 5.1.1)

**Definition:** A state is **senescent** iff its entry action contains **zero `generate ... to self` statements** (immediate or delayed). Generates to *other* instances do not affect the classification. `cancel`, `bridge`, attribute writes, etc. are irrelevant. The rule is purely syntactic — any occurrence of self-generation in the entry action AST, even guarded by an `if`, classifies the state as transient (conservative; no flow analysis required).

**Compiler responsibilities:**
- For each state in each class state machine, walk the parsed entry action AST and check for any `generate <event> to self` (with or without `delay`) statements
- Populate the `senescent_states: set[str]` field on each `ClassManifest` with the names of states that pass the test
- The field is added to `engine/manifest.py` in Phase 5.1.1, so by the time Phase 5.2 ships the runtime can already consume it
- Add a Phase 5.2 unit test: a hand-built model with known transient/senescent states verifies the classifier
- Acceptance against the elevator model: `Idle`, `Moving`, `Exchanging` must classify as senescent; `Departing`, `Floor_Updating`, `Arriving` must classify as transient (per `docs/MODEL-BUILDING-PROCESS.md:73-77`)

**Why static, not runtime:** The runtime can detect "instance happens to be at rest right now" but cannot tell whether that's a *modeled* resting state or a *temporary* gap in a transient state's conditional self-generation. Static classification gives the modeler a definitive answer that matches the design intent in the YAML.

**Why pure syntactic, not flow-analyzed:** Flow analysis would require interpreting guard conditions and predicting branches — out of scope and brittle. The conservative rule ("any syntactic self-generate makes it transient") ships now; a flow-aware classifier could land in a later phase if needed.

## What's Out of Scope for Phase 5.2

These belong to later phases — capture as "deferred ideas":
- **Bundle loader** — Phase 5.3 (loads `.mdfbundle` and instantiates the runtime)
- **MCP tool wrapping** (`compile_domain` exposed via MCP server) — Phase 5.3
- **Multi-domain compilation** — single bundle per domain in 5.2; multi-domain in a later milestone
- **Incremental compilation / caching** — full recompile every time in 5.2
- **Bundle signing / encryption** — bundles are opaque but not secret in 5.2
- **Vendoring `engine/` into the bundle** — deferred until we actually deploy bundles outside the source tree

## Inputs Researcher Should Investigate

1. **Lark `Transformer` patterns** — best practices for emitting source from a parse tree, especially with `pycca/grammar.py`'s existing Earley-for-guards / LALR-for-statements split
2. **Elevator model action coverage** — read `examples/elevator/`'s YAML and pycca files; produce a checklist of every Tier 1/2/3 construct it actually uses, so SC-07 acceptance is testable from day one
3. **Source location preservation** — how to thread `(file, line, col)` from Lark tokens into generated code comments
4. **Black integration** — invoke as library or subprocess; failure modes
5. **Determinism gotchas** — Python set iteration, dict ordering of nested structures, JSON serialization stability for `bundle.json` / `manifest.json`
6. **Refinement type encoding** — how MDF refinement constraints are currently expressed in `schema/yaml_schema.py`, and the cleanest way to translate them to runtime validators
7. **Existing pycca grammar gaps** — does `pycca/grammar.py` cover the entire Tier 3 surface, or are there constructs documented in SYNTAX.md but not yet parseable? Any gaps become Phase 5.2 grammar work.
