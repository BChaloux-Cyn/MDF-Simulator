## Summary

This phase is an internal engine improvement with no new public API surface. All 8 items from ENG-001 are fully specified in CONTEXT.md with locked decisions. The work is pure Python refactoring and extension of existing frozen dataclasses, the scheduler loop, the clock, and the codegen pipeline.

The main risk area is **D-02 Option A**: the codegen change from a single `Callable` to `list[tuple[str, Callable]]` touches three layers simultaneously ŌĆö `compiler/codegen.py` (_render_action_fn), `compiler/transformer.py` (transform_action), `engine/scheduler.py` (_invoke_action / ActionExecuted yield sites), and `engine/bundle_loader.py` (rebinding logic). Every existing test that constructs `TRANSITION_TABLE` by hand (e.g., `test_engine.py`, `test_simulation_e2e.py`) will need the `action_fn` value updated to the new list-of-tuples form.

The secondary risk is **D-07**: adding `kw_only=True` fields to a frozen base dataclass in Python 3.11+ is straightforward, but 12 subclasses each have at least one non-default positional field. Research confirms this pattern is safe at Python 3.13 (the project runtime). `dataclasses.replace()` on frozen dataclasses for stamping is idiomatic and requires no frozen-override hacks.

**Primary recommendation:** Implement in four isolated waves: (1) clock + scheduler bugs (D-13ŌĆōD-19), (2) microstep base fields + event_id (D-07ŌĆōD-11), (3) ActionExecuted context (D-01), (4) codegen Option A (D-02ŌĆōD-06) + bundle loader update.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Stamping frozen dataclass fields | `object.__setattr__` | `dataclasses.replace()` | `replace()` is the stdlib idiom for frozen dataclasses; __setattr__ bypasses the frozen contract |
| Attribute snapshot/diff | Custom deep-copy | `dict(instance)` before + dict comprehension after | Instance dicts are shallow strŌåÆscalar; `dict()` copy is sufficient and correct |
| Sleeping in real-time mode | `asyncio.sleep` / `threading.Timer` | `time.sleep()` | Simulation loop is synchronous; stdlib sleep is the right primitive |
| kw_only base fields on frozen dataclass | Per-subclass workarounds | Python 3.10+ `field(kw_only=True)` | The project already requires Python 3.11+; kw_only fields on a parent work cleanly |

### transform_action ŌĆö current state

`compiler/transformer.py` lines 854ŌĆō886: [VERIFIED: read source]

## Common Pitfalls

### Pitfall 1: Forgetting `event_id` default on `Event` breaks all existing construction sites

**What goes wrong:** Adding `event_id: int` without a default makes `Event(...)` calls positional-argument-incompatible ŌĆö all test fixtures and scheduler internals that construct `Event` explicitly will raise `TypeError`.

**How to avoid:** Add `event_id: int = 0` with a default. The scheduler overwrites it via `dataclasses.replace(event, event_id=counter)` immediately after construction at enqueue time. Existing construction sites stay unchanged.

### Pitfall 2: snapshot/diff must copy before, not after

**What goes wrong:** Taking the snapshot after calling the callable gives an empty diff (before == after). The snapshot must be `pre = dict(instance)` before calling the callable, then `assignments_made = {k: v for k, v in instance.items() if pre.get(k) != v}`.

**Warning sign:** All `assignments_made` dicts are empty in the trace even for states that assign attributes.

### Pitfall 3: `advance_to()` applying `speed_multiplier` to the time-set

**What goes wrong:** Multiplying `ms` by `speed_multiplier` when setting `self._time_ms` causes the clock to jump past expiry events. The multiplier is a real-time scaling factor, not a model-time factor.

**How to avoid:** `self._time_ms = max(self._time_ms, ms)` ŌĆö no multiplier. The sleep duration is `(ms - current) / (speed_multiplier * 1000)`.

### Pitfall 4: position tracking offset in pycca source recovery

**What goes wrong:** `child.meta.start_pos` is a character offset from the start of the full action source string. If the full source is passed as-is, `source[start:end]` is correct. If it's been stripped or prefixed, the offsets are wrong.

**How to avoid:** Pass the raw `action_src` string (before any transformation) to the parser, then use character slices directly from that string.

### Pitfall 5: `test_engine.py` hand-rolled `action_fn` type mismatch

**What goes wrong:** After D-02, `_invoke_action` is no longer called with a single callable from `action_fn`. If `test_engine.py` still sets `action_fn: _noop_action`, the scheduler iterates `_noop_action` as if it's a list, raising `TypeError`.

**How to avoid:** Update all hand-rolled `TRANSITION_TABLE` entries to use `action_fn: [("pass", _noop_action)]`.

### Pitfall 6: `_enqueue_delay` constructs a new `Event` without `event_id`

**What goes wrong:** In `tick_delay_queue()`, the unwrapped event is reconstructed as a new `Event(...)` without carrying forward `event_id` (line 250ŌĆō259 of `scheduler.py`). The `EventDelayExpired` and the downstream `EventReceived` would lose the ID linkage.

**How to avoid:** When constructing the unwrapped event in `tick_delay_queue()`, carry forward the original `event_id`: `Event(..., event_id=event.event_id)`.

### Stamping loop in execute() (D-08)

```python
# In execute(), wrapping every yield:
_step_counter: int = 0
_start_ms: float = time.monotonic() * 1000.0

def _stamp(step: MicroStep) -> MicroStep:
    nonlocal _step_counter
    ts = time.monotonic() * 1000.0 - _start_ms
    s = dataclasses.replace(step, step_id=_step_counter, timestamp_ms=ts)
    _step_counter += 1
    return s
```
All `yield step` sites become `yield _stamp(step)`.

### advance_to() implementation (D-15)

```python
import time

def advance_to(self, ms: float) -> None:
    """Jump simulation time to ms; sleep in real-time mode (D-15)."""
    if not self._fast_forward:
        delta = ms - self._time_ms
        if delta > 0:
            time.sleep(delta / (self._speed_multiplier * 1000.0))
    self._time_ms = max(self._time_ms, ms)
```

### Per-statement codegen (D-02, Option B)

```python
# In _render_action_fn (replacement logic):
def _render_action_stmts(fn_name_prefix: str, body_src: str, source_file: str, source_line: int) -> str:
    """Emit a list[tuple[str, Callable]] assignment for one action slot."""
    if not body_src or not body_src.strip():
        # D-06: empty body ŌåÆ single pass sentinel
        return f'{fn_name_prefix} = [("pass", lambda ctx, self_dict, params: None)]'

    parser = _make_action_parser()
    tree = parser.parse(body_src)
    pairs: list[str] = []
    for stmt_tree in tree.children:
        start = stmt_tree.meta.start_pos
        end = stmt_tree.meta.end_pos
        pycca_src = body_src[start:end].strip()
        transformer = ActionTransformer()
        py_body = transformer.transform(stmt_tree)
        fn_body = textwrap.indent(py_body.strip() or "pass", "    ")
        fn_literal = (
            f"lambda ctx, self_dict, params: (\n{fn_body}\n    None\n)"
        )
        pairs.append(f'    ({pycca_src!r}, {fn_literal})')
    pairs_str = ",\n".join(pairs)
    return f'{fn_name_prefix} = [\n{pairs_str}\n]'
```

Note: lambdas cannot contain statements (only expressions). For statement bodies, a named inner function per statement is safer than a lambda. The codegen should emit named functions (`def _stmt_N_<fn_name>(ctx, self_dict, params): ...`) and reference them in the list.

## Open Questions

1. **Lambda vs named inner function for per-statement callables**
   - What we know: Python `lambda` bodies can only contain a single expression, not statements (`if`, `for`, assignments with side effects). The transformed pycca output is Python statements, not expressions.
   - What's unclear: D-04 locks the signature `(ctx, self_dict, params) -> None` but does not specify lambda vs named def for each per-statement callable.
   - Recommendation: Use named inner functions (`def _stmt_N_...`) rather than lambdas. This preserves readable `__name__` attributes and avoids the expression-only lambda constraint. The D-04 signature applies to the named function form as well.

2. **`tick_delay_queue` event_id forwarding**
   - What we know: `tick_delay_queue` constructs a new `Event(...)` for each expired event (line 250ŌĆō259 of `scheduler.py`).
   - What's unclear: D-11 does not explicitly mention `tick_delay_queue` as a propagation site.
   - Recommendation: The new unwrapped event must carry forward `event.event_id` to maintain ID continuity from `EventDelayed` through `EventDelayExpired` to `EventReceived`.
