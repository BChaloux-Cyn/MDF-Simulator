## Per-statement trace (Item 2)

| Option | Description | Selected |
|--------|-------------|----------|
| Option B: `__pycca_lines__` annotation | Annotate action function post-execution; emit one record per line after function runs. Lower risk, no codegen refactor. | |
| Option A: List of `(pycca_src, callable)` pairs | Each pycca statement is a separate callable; scheduler executes one-by-one, yielding one `ActionExecuted` per step. | ✓ |

**User's choice:** Option A — initially chose Option B, then clarified that pause-at-line capability is required. Option B cannot pause mid-execution (whole function runs before any records emitted), so Option A is necessary.

**assignments_made sub-question:**

| Option | Description | Selected |
|--------|-------------|----------|
| `ctx.assign()` records it | ctx.assign() tracks the last assignment; scheduler reads it after each callable. | |
| Snapshot before/after | Scheduler snapshots instance attributes before and after each callable and diffs them. | ✓ |

**Empty action sub-question:**

| Option | Description | Selected |
|--------|-------------|----------|
| No record | Emit nothing for a pass action. | |
| One 'pass' record | Emit `ActionExecuted(pycca_line='pass', assignments_made={})`. | ✓ |

**Notes:** User confirmed that 5 actions currently compiled as `pass` are intentionally empty (Elevator.Idle, Elevator.Moving, FloorCallButton.Idle, DestFloorButton.Idle, DestFloorButton.Lit — no `entry_action` in YAML). 5 others (CarDoor.Opening/Closing, Elevator.Arriving/Exchanging, FloorCallButton.Lit) are stale bundles — pycca exists in YAML and transformer supports the patterns; bundles need regeneration.

## Fast-forward / Clock modes (Item 8)

| Option | Description | Selected |
|--------|-------------|----------|
| Real-time in 5.4/5.5 only | Only fast-forward in this phase. | |
| Both modes now | `fast_forward` flag; scheduler sleeps in real-time mode. | ✓ |

**Sleep location:**

| Option | Description | Selected |
|--------|-------------|----------|
| Inside `clock.advance_to()` | Clock encapsulates timing; scheduler always calls `advance_to()` the same way. | ✓ |
| In `scheduler.execute()` loop | Scheduler checks mode and sleeps itself. | |

**Default:**

| Option | Description | Selected |
|--------|-------------|----------|
| `fast_forward=True` | Discrete-event by default; tests run instantly. | |
| `fast_forward=False` | Real-time by default; speed_multiplier scales wall time. | ✓ |

**Notes:** User clarified that the default should be real-time with `speed_multiplier=1000.0` — a 10s modeled delay takes 10ms of real time. This is "fast enough for tests" while preserving real-time semantics. `advance_to()` uses `max(current, target)` and applies multiplier only to sleep duration (not the time-set). Tests use a `fast_clock` pytest fixture with `fast_forward=True`.
