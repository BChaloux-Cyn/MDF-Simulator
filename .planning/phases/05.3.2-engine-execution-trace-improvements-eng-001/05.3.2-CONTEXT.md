# Phase 05.3.2: engine-execution-trace-improvements (ENG-001) — Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix two scheduler bugs (early termination when delayed events are pending, no real-time clock advance), and improve the micro-step trace format with class/instance context, per-statement granularity, step IDs, timestamps, and event ID linking. All eight items from `issues/engine-execution-trace-improvements.md` are in scope.

Also in scope: regenerate the elevator bundles to pick up Phase 05.3.1 transformer fixes (several action bodies were compiled as `pass` due to stale bundles — `CarDoor.Opening/Closing`, `Elevator.Arriving/Exchanging`, `FloorCallButton.Lit`).

</domain>

<decisions>
## Implementation Decisions

### Item 1 — ActionExecuted class/instance context
- **D-01:** Add `class_name: str` and `instance_id: dict` fields to the `ActionExecuted` dataclass. Both call sites in `scheduler.py` (lines ~458 and ~524) have `concrete_class` / `concrete_id` in scope — pass them directly.

### Item 2 — Per-statement ActionExecuted (Option A)
- **D-02:** Implement **Option A**: codegen emits each pycca statement as a `(pycca_src, callable)` pair. The action entry in `TRANSITION_TABLE` changes from `action_fn: Callable` to `action_fn: list[tuple[str, Callable]]`. The scheduler iterates the list and executes each callable, yielding one `ActionExecuted` per statement.
- **D-03:** `pycca_src` in each tuple is the original pycca source text for that statement (from the AST at codegen time).
- **D-04:** Each callable signature remains `(ctx: SimulationContext, self_dict: dict, params: dict) -> None` — same as the current monolithic function signature.
- **D-05:** `assignments_made` is populated via **snapshot before/after**: the scheduler snapshots the instance's attribute dict before calling the callable, runs it, then diffs the snapshot against the post-call state. The diff becomes `assignments_made` for that `ActionExecuted` record.
- **D-06:** Actions with no pycca statements (no `entry_action` field in YAML, or truly empty bodies) emit a **single `ActionExecuted` record** with `pycca_line="pass"` and `assignments_made={}`. This makes it visible in the trace that the action slot was invoked.

### Item 3+4 — Timestamps and Step IDs
- **D-07:** Add `step_id: int` and `timestamp_ms: float` to the **`MicroStep` base dataclass** using `field(default=0, kw_only=True)` and `field(default=0.0, kw_only=True)`. Python 3.11 `kw_only=True` resolves the frozen dataclass inheritance ordering issue — subclasses with non-default positional fields continue to work.
- **D-08:** The scheduler stamps `step_id` and `timestamp_ms` via `dataclasses.replace(step, step_id=counter, timestamp_ms=ts)` before yielding each step to the consumer. `step_id` is a sequential counter starting at 0 per simulation run. `timestamp_ms` is monotonic time in ms from simulation start.
- **D-09:** No trace format versioning needed — no external consumers of `.design/traces/*.json` exist yet.

### Item 5 — Event IDs
- **D-10:** Add `event_id: int` to the `Event` dataclass. Assigned via a monotonic `_event_id_counter` in the scheduler when an event is enqueued (both immediate and delay paths).
- **D-11:** Propagate `event_id` to four microstep types: `GenerateDispatched`, `EventDelayed`, `EventReceived`, `SchedulerSelected`. This links every record related to one event dispatch.

### Item 6 — Queue classification for subtype cross-instance events
- **D-12:** Verify that `_classify_queue` in `scheduler.py` checks both `sender_class == target_class` AND `sender_id == target_id` for self-directed routing. Write a test that explicitly verifies CarDoor→ShaftDoor events (same `door_id` dict value, different classes) are classified as `standard`, not `priority`.

### Item 7 — Early termination fix
- **D-13:** In `execute()`, replace the bare `else: break` with:
  ```python
  elif self._delay:
      next_expiry, _, _ = self._delay[0]
      self._clock.advance_to(next_expiry)
      # loop continues — tick_delay_queue() drains it
  else:
      break
  ```

### Item 8 — Clock modes (fast-forward + real-time)
- **D-14:** Add `fast_forward: bool` parameter to `SimulationClock.__init__`. Default: `fast_forward=False` (real-time mode with accelerated wall clock).
- **D-15:** Add `advance_to(ms: float)` method to `SimulationClock`. Behavior:
  - Both modes: sets `self._time_ms = max(self._time_ms, ms)` after the wait (exact target, no multiplier applied to the time-set itself).
  - `fast_forward=True`: no sleep — jumps immediately.
  - `fast_forward=False`: sleeps `(target_ms - current_ms) / (speed_multiplier * 1000)` real seconds before setting the time. `speed_multiplier` acts as a wall-clock accelerator.
- **D-16:** Default `speed_multiplier` changes from `1.0` to `1000.0`. With `fast_forward=False, speed_multiplier=1000.0`, a 10s modeled delay takes 10ms of real time — fast enough for tests while preserving real-time semantics.
- **D-17:** GUI debugger uses `SimulationClock(fast_forward=False, speed_multiplier=1.0)` for full real-time playback. Tests that need pure discrete-event speed use `SimulationClock(fast_forward=True)`.
- **D-18:** The sleep happens **inside `clock.advance_to()`** — the clock encapsulates timing behavior. The scheduler's `execute()` always calls `advance_to()` the same way regardless of mode.
- **D-19:** A pytest fixture (`fast_clock` or similar) providing `SimulationClock(fast_forward=True)` is added to `tests/conftest.py`. Existing delay-based scenario tests use this fixture.

### Bundle Regeneration
- **D-20:** Elevator bundles must be regenerated as part of this phase's test/verification step. The stale bundles have 5 action functions compiled as `pass` that the transformer can already handle (`CarDoor.Opening/Closing`, `Elevator.Arriving/Exchanging`, `FloorCallButton.Lit`). Regeneration is a test prerequisite, not new compiler work.

<code_context>
## Existing Code Insights

### Reusable Assets
- `engine/microstep.py`: All frozen dataclasses. Python 3.11 `kw_only=True` field modifier is the mechanism to add `step_id`/`timestamp_ms` to the base without breaking subclass field ordering.
- `engine/scheduler.py`: `_write_trace` / `_step_to_dict` serialization — `dataclasses.asdict()` will automatically include new base class fields once they exist.
- `SimulationClock`: Already a pure internal-counter clock with `speed_multiplier`, `pause()`, `resume()`. `advance_to()` is an additive method.

### Established Patterns
- Frozen dataclasses throughout `engine/` — use `dataclasses.replace()` to produce stamped copies; avoid `object.__setattr__`.
- `kw_only=True` per-field (Python 3.10+): `field(default=0, kw_only=True)` on base class fields avoids MRO ordering conflicts with non-default subclass fields.
- `TRANSITION_TABLE` in generated Python files: `dict[tuple[str,str], dict]` — `action_fn` value type changes from `Callable` to `list[tuple[str, Callable]]`.

### Integration Points
- `scheduler._process_event` and `_process_creation` — the two code paths that call `action_fn` and yield `ActionExecuted`. Both need updating for Option A iteration and snapshot logic.
- `bundle_loader.py` — rebinds action functions from generated modules into `ClassManifest`. Must handle the new `list[tuple[str, Callable]]` type.
- `tests/conftest.py` — add `fast_clock` fixture here.

</code_context>

<specifics>
## Specific Ideas

- The wall-clock accelerator (`speed_multiplier=1000.0` default) is the primary way to make tests fast while keeping real-time semantics. `fast_forward=True` is for pure discrete-event speed (no timing at all).
- `advance_to(ms)` uses `max(self._time_ms, ms)` to prevent going backwards — important for correctness in case of any reordering edge.

</specifics>
