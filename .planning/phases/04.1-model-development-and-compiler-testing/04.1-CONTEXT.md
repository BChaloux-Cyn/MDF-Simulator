<decisions>
## Implementation Decisions

### Sequence

Three tracks in strict order:
1. **Schema fixes** — repair broken schema artifacts so validate_model can run cleanly
2. **Model completeness** — resolve all open issues, complete the elevator model structure
3. **Action body authoring + compiler testing** — author all pycca bodies (single agent, sequential), fix grammar gaps as found, write tests

### Schema fixes (blocking — must be first)

Three things are currently broken:
- `validation.py`: two `formalizes` validation blocks reference `cls.formalizes`, which no longer exists on `ClassDef` (causes `AttributeError`). Remove both blocks (~lines 215–221 and ~918–924).
- `tests/test_yaml_schema.py`: test fixture uses `stereotype: associative` which is no longer valid. Remove the associative test data and the `test_associative_missing_formalizes_rejected` test.
- `FloorIndicator` in the elevator model: still declares `stereotype: associative` and `formalizes: R10` — will fail schema validation. Migrate per the decision below.

### FloorIndicator migration

Replace the many-to-many R10 (Elevator ↔ Floor) + associative FloorIndicator with:
- `FloorIndicator`: **active** class (has state machine, reacts to floor change events)
- **R10**: FloorIndicator (1) ↔ Elevator (0..*) — "shows floor for" / "has indicator at"
- **R13**: FloorIndicator (1) ↔ Floor (0..*) — "is located on" / "has indicator"
- FloorIndicator holds referentials: `r10_elevator_id` (UniqueID) and `r13_floor_num` (FloorNumber)
- The old R10 M:M association is removed from `class-diagram.yaml`

### Issue resolution (all 5 in-phase)

- **ELV-001** (subtype inherits supertype referential attributes): Subtypes should NOT re-declare identifier attributes that exist on the supertype. Fix via schema-level inheritance — `yaml_schema.py` propagates supertype identifier attributes to subtypes automatically. `ElevatorCall` and `FloorCall` drop their redundant `call_id` declarations.
- **ELV-003** (linked list via R9 — metamodel support): Claude's discretion — investigate whether this requires a yaml_schema.py change or is resolvable as a modeling pattern. If no schema change is needed, document it as the canonical queue idiom in the BoK (Phase 4.2). Log findings in the issue file.
- **ELV-005** (At_Floor entry generates to class name, not instance): Fix in elevator model YAML — generate statement targets an instance, not a class name.
- **ELV-006** (Door_closed event routed to wrong recipient): Fix in elevator model YAML.
- **ELV-007** (Dispatcher has no association link to pending FloorCall queue): Fix in elevator model YAML — add the missing association.

### Action body authoring

- All **9 active classes** get complete, simulate-ready pycca action bodies: Elevator, Door, Dispatcher, Floor, ElevatorCall, FloorCall, Request, Shaft, ElevatorIndicator, FloorIndicator (10 including the migrated FloorIndicator)
- No null placeholders in state entry/exit or transition action fields when done
- **Single agent, sequential** — author one class at a time; observe and note patterns for Phase 4.2 BoK
- **Grammar gaps**: if a pycca construct is needed but the Lark grammar doesn't support it, fix `pycca/grammar.py` immediately and continue. Phase 4.1 exits with a grammar that parses the full elevator model.

### Compiler testing

Two test files:
- `tests/test_elevator.py` — end-to-end: run `validate_model` on the elevator domain; assert zero issues. This exercises the full pipeline including pycca syntax pre-check.
- `tests/test_pycca.py` — unit: call `STATEMENT_PARSER.parse()` and `GUARD_PARSER.parse()` directly on representative pycca body strings covering every construct type used in the elevator model.

### Claude's Discretion

- How ELV-003 (linked list R9) is resolved — schema change vs. documented pattern
- Exact FloorIndicator state machine design (states, transitions, events)
- Order of class authoring within the sequential pass
- Grammar fix approach when gaps are found (extend existing rules vs. new rules)
