## Summary

Phase 04.1 is a repair-and-completion phase with three strictly ordered tracks. The phase
has no external dependencies — all the information needed to plan it is entirely within
the repository.

**Track 1 (Schema fixes)** has three precisely located breakage points: two `cls.formalizes`
references in validation.py (lines 215 and 918) that will cause AttributeError because
`ClassDef.formalizes` was removed from yaml_schema.py, one test fixture in test_yaml_schema.py
that uses the now-invalid `associative` stereotype, and the FloorIndicator class in the
elevator model that still uses `stereotype: associative`.

**Track 2 (Model completeness)** has five open issues with root causes confirmed by reading
the actual YAML files. ELV-001, ELV-005, ELV-006, and ELV-007 are pure model edits. ELV-003
requires a design decision about the Request linked-list head pointer; the issue file
recommends Option A (explicit association), which is the right approach given Phase 5 needs
a clean model.

**Track 3 (Action bodies + compiler testing)** requires reading all 10 active class state
diagrams and identifying which action body slots are missing or malformed, then authoring
complete pycca bodies. The grammar has 8 known gaps (documented in SYNTAX.md section 7)
that will need to be closed before the test suite can pass.

**Primary recommendation:** Execute the three tracks in strict order with no overlap. Schema
fixes are a prerequisite for any validate_model call; model completeness must precede action
body authoring because the structural issues (ELV-007, FloorIndicator migration) affect which
relationships action bodies can traverse.

## Architecture Patterns

### Established Patterns (from STATE.md + code inspection)

**Model root isolation in tests:**
```python

def test_elevator(monkeypatch, tmp_path):
    monkeypatch.chdir("examples/elevator")
    issues = validate_model("Elevator")

```
`MODEL_ROOT` in validation.py is anchored to CWD. Tests must `monkeypatch.chdir` to
the model root before calling `validate_model`.

**Never raise — always return issues list:**
```python
# Source: tools/validation.py pattern
def validate_model(domain: str) -> list[dict]:
    issues = []
    # ... all checks append to issues, never raise
    return issues
```
The test_elevator.py pattern is `assert issues == []`, not a try/except.

**State diagram YAML field names:**
```yaml

  - name: SomeName
    entry_action: "pycca code here;"   # str | None

  - from: StateA
    to: StateB
    event: EventName
    guard: "expr"      # str | None
    action: "pycca;"   # str | None
```
Fields `entry_action`, `action`, and `guard` are all `str | None`.

**Guard/statement parser usage:**

## Common Pitfalls

### Pitfall 1: `cls.formalizes` AttributeError Causes Validator to Crash

**What goes wrong:** `validation.py` lines 215 and 918 reference `cls.formalizes` / `cls_def.formalizes`.
`ClassDef` no longer has a `formalizes` field (`stereotype: associative` was removed). Any call
to `validate_model` or `validate_class` on a model that has any class will trigger an `AttributeError`
and the function will raise instead of returning an issues list.

**Location 1:** `validation.py` lines 215–221 (inside `_check_class_referential_integrity`):
```python
if cls.formalizes is not None and cls.formalizes not in assoc_names:
    issues.append(...)
```

**Location 2:** `validation.py` lines 918–924 (inside `validate_class`):
```python
if cls_def.formalizes is not None and cls_def.formalizes not in assoc_names:
    issues.append(...)
```

**Fix:** Delete both if-blocks in their entirety. No replacement code is needed — the
`formalizes` concept is gone.

**Warning signs:** Any test calling `validate_model` or `validate_class` that produces
`AttributeError: 'ClassDef' object has no attribute 'formalizes'`.

---

### Pitfall 2: FloorIndicator Migration Has Two Referential Naming Concerns

**What goes wrong:** The current FloorIndicator class has two referentials both labeled
`referential: R10`. After migration, they become separate associations R10 and R13.
The naming convention from COMPILATION.md is `r<N>_<target_identifier_attr>`. The
identifier of Elevator is `elevator_id`, so the referential becomes `r10_elevator_id`.
The identifier of Floor is `floor_num`, so the referential becomes `r13_floor_num`.

The existing FloorIndicator.yaml state machine references `r10_elevator_id` and
`r10_floor_num`. After migration, the `r10_floor_num` referential changes to `r13_floor_num`.
The FloorIndicator.yaml state machine does not directly reference either in its current
(incomplete) action bodies, but any action bodies authored must use the new names.

**Warning signs:** Action bodies that reference `r10_floor_num` will be wrong after migration.

---

### Pitfall 3: Grammar Currently Does Not Parse Most Real Pycca Constructs

**What goes wrong:** The grammar in `pycca/grammar.py` (as confirmed by SYNTAX.md section 7)
does NOT support the following constructs that appear throughout the elevator state diagrams:

| Gap | Used in elevator model |
|-----|----------------------|
| `select ... related by self->R<N>` | Elevator.yaml, Shaft.yaml, FloorCall.yaml, etc. |
| `generate Event to <variable>` (non SELF/CLASS target) | Elevator.yaml (At_Floor), Floor.yaml, etc. |
| `generate Event(param: val) to ...` (params) | Shaft.yaml (Floor_reached), etc. |
| `relate <var> to <var> across R<N>` | Will be needed in Dispatcher action bodies |
| `unrelate` | Will be needed in cleanup actions |
| `elif` / `else` | Elevator.yaml (Idle entry action) |
| `delete <var>` (variable form, not `delete object of`) | Request terminal state cleanup |
| `var.attribute` reads (non-self) | Elevator.yaml (At_Floor: `next_req.destination_floor`) |
| Arithmetic (`a + b`, `a - b`) | Not yet used but defined in SYNTAX.md |

**Warning signs:** Any test calling `STATEMENT_PARSER.parse()` on real elevator action
body strings will fail with `UnexpectedInput` until the grammar is extended.

**Fix approach:** Extend the PYCCA_GRAMMAR string in grammar.py. Add new rules for
relationship traversal select, `else`/`elif` clauses, variable-target generate with/without
params, relate/unrelate, delete variable form, and `var.attr` reads. Re-test after each
addition.

### Pitfall 4: Existing Elevator Action Bodies Have Real Errors (Not Just Incomplete)

Several current action bodies contain confirmed errors beyond "incomplete":

**Shaft.yaml Moving_Up/Moving_Down `Floor_sensor_triggered` action:**
```yaml
action: >-
  self.current_floor = floor_num;
  select any elev from instances of Elevator where R11;
  generate Floor_reached to elev;
```
- `self.current_floor = floor_num;` — `floor_num` is a bare name. Per SYNTAX.md name
  resolution, bare names must be locally bound or enum values. `floor_num` is an event
  parameter — it must be `rcvd_evt.floor_num`. This is a modeling error.
- `select any elev from instances of Elevator where R11;` — `R11` in a `where` clause
  is not valid `select ... from instances of ... where <boolean_expr>` syntax. Relationship
  traversal uses `select any elev related by self->R11;` instead.
- `generate Floor_reached to elev;` — `Floor_reached` event has a `floor_num` parameter
  but none is passed. Must be `generate Floor_reached(floor_num: self.current_floor) to elev;`.

**Elevator.yaml Idle entry action:**
```yaml
entry_action: >-
  select any req from instances of Request where r2_elevator_id == self.elevator_id;
  if req != empty; generate Request_assigned to SELF;
  else; generate Elevator_available to Dispatcher; end if;
```
- `generate Elevator_available to Dispatcher;` — `Dispatcher` is a bare class name. Must
  use a `select` to bind an instance, then generate to the variable.
- `generate Request_assigned to SELF;` — `Request_assigned` has a `target_floor` parameter
  but none is passed.
- The `if ... else ... end if` block uses semicolons rather than braces — this is the
  current grammar's syntax (`if expr; stmts end if;`), but the grammar does not yet support
  `else`. The else branch will require grammar extension.

**Elevator.yaml Moving_Up/Moving_Down entry actions:**
```yaml
entry_action: >-
  select any shaft from instances of Shaft where R11;
  generate Move_up to shaft;
```
Same problem: `where R11` is not valid select-from-instances syntax; must be `related by`.

**Door.yaml Closed entry action (ELV-006):**
```yaml
entry_action: "generate Door_closed to r1_elevator_id;"
```
`Door_closed` is not in Elevator's event list. This is the ELV-006 routing bug.

**ElevatorCall.yaml Fulfilled entry action (ELV-001):**
```yaml
entry_action: "generate Call_fulfilled to r7_button_id;"
```
`r7_button_id` is a relvar on the supertype `Call` accessed via R7. This is ELV-001.
The `r7_button_id` naming follows the `r<N>_<target_id>` convention from COMPILATION.md
and should be valid once subtype inheritance is implemented.

**Dispatcher.yaml transition actions (ELV-007):**
```yaml
action: "create object of Request; generate Request_assigned to Elevator;"
```
- `create object of Request;` — old grammar syntax. New syntax is `create req of Request;`.
- `generate Request_assigned to Elevator;` — `Elevator` is a class name, not an instance.

### Pitfall 5: ELV-003 Must Be Resolved Before ELV-007

The `Elevator` class has `r_queue_head_id referential: R2` which is wrong (R2 is the
ownership association, not a head-pointer). ELV-003 must be resolved (by adding an explicit
head-pointer association) before ELV-007 action body authoring can correctly reference
the queue head. The recommended fix (Option A from the issue file) is to add a new
association — this will likely consume an available R number such as R14 or R15.

---

### Pitfall 6: R5 Partition Only Declares ElevatorCall Side

The current class-diagram.yaml has:
```yaml
- name: R5
  point_1: Call
  point_2: ElevatorCall
  ...
```
R5 only covers the Call ↔ ElevatorCall leg. FloorCall specializes R5 via `specializes: R5`
but R5 as currently defined does not include FloorCall as point_2. The validator will flag
this (class specializes R5 but is not listed in supertype's partitions). The `Call` class
has a partition listing both ElevatorCall and FloorCall, but there is only one R5 association
entry. Generalization partitions need careful alignment.

---

### Pitfall 7: `else` Clauses in Entry Actions Need Grammar Support

Current grammar `if_stmt` rule:
```
if_stmt: "if" expr ";" statement* "end" "if" ";"
```
There is no `else` or `elif` branch. Multiple existing entry actions use `else` (Elevator.yaml
Idle, At_Floor). The grammar extension for else is: add an optional `"else" ";" statement*`
clause inside the if_stmt rule.

## Grammar Extension Guide

The gaps in `pycca/grammar.py` that must be closed during Track 3, in recommended extension order:

| Priority | Gap | New Grammar Rule Sketch |
|----------|-----|------------------------|
| 1 | `else`/`elif` in if_stmt | Add optional `("else" ";" statement*)?` to if_stmt |
| 2 | `select related by self->R<N>` | New `select_related_stmt` rule with `"related" "by" NAME "->" NAME` |
| 3 | `generate Event to <var>` | Change generate target from `("SELF" \| "CLASS")` to `(NAME)` |
| 4 | `generate Event(param: val) to <var>` | Add optional param list to generate_stmt |
| 5 | `var.attribute` reads | New atom variant `NAME "." NAME` |
| 6 | `relate`/`unrelate` | New `relate_stmt`, `unrelate_stmt` rules |
| 7 | `delete <var>` (variable form) | Extend delete_stmt to accept `"delete" NAME ";"` |
| 8 | Create with params: `create var of Class(p: v)` | Extend create_stmt |
| 9 | Arithmetic `a + b` | Add arithmetic rules to expr |

The existing `generate_stmt` points `to ("SELF" | "CLASS")` — these are uppercase constants,
not the variable-name form. The current elevator model uses lowercase `to elev`, `to shaft`,
etc. Extend to `to NAME` where NAME covers both.

### ELV-001 Fix (yaml_schema.py subtype inheritance)

The validator needs to compute the "effective attribute set" for a subtype by merging
supertype identifier attributes into the subtype during referential integrity checks.
The Pydantic model itself does not need to change — only the validation logic needs to
look up the supertype's attributes when checking a subtype.

```python
# Source: design from issue file + COMPILATION.md section 8
# In validation _check_class_diagram or a new helper:
def _get_effective_attributes(cls: ClassDef, class_map: dict) -> list[Attribute]:
    if cls.specializes is None:
        return cls.attributes
    supertype = class_map.get(...)  # find class that has partition for cls.specializes
    return supertype.attributes + cls.attributes
```

## Open Questions

1. **ELV-003: Option A vs Option B for linked list head pointer**
   - What we know: `r_queue_head_id referential: R2` on Elevator is wrong; R2 is the ownership association, not a head pointer
   - What's unclear: Whether to add a new explicit association (Option A) or use engine-managed linked list (Option B)
   - Recommendation: Option A (explicit association, e.g., R14: Elevator 0..1 -- 0..1 Request "has as queue head" / "is queue head for"). This keeps the model self-describing without requiring Phase 5 engine primitives, and is consistent with how COMPILATION.md documents self-referential associations.

2. **ELV-006: Which routing option for Door_closed?**
   - What we know: Three options documented in the issue file (A: Door → Elevator → Request; B: new Door-Request association; C: simplified Elevator logic)
   - What's unclear: Which produces the cleanest model for Phase 5 simulation
   - Recommendation: Option A (Door generates to Elevator, Elevator relays to Request). It avoids a new association, keeps Door decoupled from Request, and matches the existing Elevator event list pattern.

3. **ELV-001: Scope of yaml_schema.py change**
   - What we know: The subtype classes ElevatorCall and FloorCall currently re-declare `call_id` which exists on their supertype Call
   - What's unclear: Whether the schema change should (a) only add validation that subtypes don't redeclare supertype identifiers, or (b) also compute an "effective attribute set" used throughout validation
   - Recommendation: (a) is sufficient for Phase 4.1. (b) is Phase 5 scope. For now: remove the redundant `call_id` declarations from ElevatorCall and FloorCall YAML, and add a validator check that flags redeclaration.

4. **`if` statement syntax: semicolons vs braces**
   - What we know: Current grammar uses `if expr; stmt* end if;` (semicolons). SYNTAX.md section 1.8 shows C-style braces.
   - What's unclear: SYNTAX.md is the spec; the grammar should match the spec, not the other way around.
   - Recommendation: Extend grammar to support brace syntax (`if (expr) { stmts }`) as the canonical form, and update existing action bodies to use it. The old semicolon-based if syntax in the elevator YAMLs is a legacy artifact that should be cleaned up during action body authoring.
