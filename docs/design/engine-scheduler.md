# Engine Scheduler and Run-to-Completion Semantics

**Date:** 2026-04-09
**Status:** Current
**Scope:** `engine/scheduler.py`, `engine/ctx.py`, `engine/registry.py`, `engine/clock.py`

This document covers implementation mechanics. For the formal execution domain rules
(D-01 through D-37) see [`simulation-engine-design.md`](simulation-engine-design.md).

---

## 1. Three-Queue Architecture

`ThreeQueueScheduler` maintains three separate queues with distinct semantics.

### Priority queue (`_priority: deque[Event]`)

Receives self-directed events — cases where `sender_class == target_class` and
`sender_id == target_id`. These are checked before the standard queue on every
iteration of the execution loop (rule 6 / rule 7 in the design doc). The intent
is that a state machine's self-generated events are processed before any inbound
cross-instance traffic, preserving local sequencing.

### Standard queue (`_standard: deque[Event]`)

Receives all other immediate events:
- Cross-instance events (sender != target)
- `__creation__` events (always standard, never priority — per rule 3)
- `__deletion__` events
- Expired delay events promoted from the delay queue

### Delay queue (`_delay: list[tuple[float, int, Event]]`)

A min-heap ordered by `(expiry_ms, seq)`. Events enter here when `delay_ms > 0`.
The heap never dispatches events directly — `tick_delay_queue()` is called at the
top of every execution loop iteration and promotes expired entries to the standard
queue (rule 8).

The `seq` field is a monotone integer tie-breaker so events with equal expiry
dispatch in FIFO order.

### Queue routing summary (D-10..D-14)

```
enqueue(event):
    if event.delay_ms > 0:
        → delay queue (D-14 at-most-one invariant applied)
    elif sender == target (class + id both match):
        → priority queue
    elif __creation__ or __deletion__:
        → standard queue
    else:
        → standard queue
```

The delay check is applied before the self/cross check. A self-directed event with
`delay_ms > 0` enters the delay queue and arrives on the standard queue when it
expires — it never uses the priority path (rule 4).

### Why three queues

A single FIFO would allow a self-directed event to be deferred behind incoming
cross-instance events, breaking local sequencing guarantees. The priority queue
ensures self-directed events are always processed next. The delay queue is kept
separate because its ordering is by expiry time, not insertion order; merging it
with the standard queue would require re-sorting on every insert.

## 2. Run-to-Completion Semantics

### The `_pending_generated` buffer

Two fields on `ThreeQueueScheduler` implement run-to-completion:

- `_in_flight: Event | None` — set to the event currently being processed; `None`
  when the scheduler is idle between events.
- `_pending_generated: list[Event]` — accumulates events generated by action code
  while `_in_flight` is not `None`.

When `enqueue()` is called while `_in_flight is not None`, the event is appended
to `_pending_generated` instead of going directly to `_priority` or `_standard`.
A `GenerateDispatched` micro-step is still yielded immediately (for observability),
but the event does not become eligible for dispatch until the current transition
finishes.

Delayed events bypass this buffer: if `delay_ms > 0`, `_enqueue_delay()` is
called regardless of `_in_flight` state, because the delay queue is time-governed
and has its own at-most-one invariant.

### Execution loop step-by-step

The following walk covers `_process_event()`, the heart of the scheduler.

1. **Delay tick.** `tick_delay_queue()` runs at the top of every `execute()` loop
   iteration. Any event whose `expiry_ms <= clock.now()` is popped from the
   min-heap, stripped of `delay_ms`, and appended to `_standard`. An
   `EventDelayExpired` micro-step is yielded.

2. **Queue selection.** Priority queue is checked first. If non-empty, the front
   event is popped and a `SchedulerSelected(queue="priority", ...)` micro-step is
   yielded. Otherwise standard queue is tried. If both are empty, `execute()` exits
   (domain idle, rule 9).

3. **Lifecycle dispatch.** `_process_event()` delegates `__creation__` and
   `__deletion__` events to dedicated handlers before doing any table lookup. See
   sections 5 and 6.

4. **Target resolution.** For a regular event, `_resolve_target()` looks up the
   concrete instance. If the primary lookup misses, subtypes are tried in order
   (polymorphic dispatch — see section 3). If no instance is found, an
   `ErrorMicroStep(error_kind="unknown_target")` is yielded and the event is
   dropped.

5. **State lookup.** `registry.get_state(concrete_class, concrete_id)` returns the
   instance's `curr_state`. An `EventReceived` micro-step is yielded.

6. **Transition table lookup.** `cdef["transition_table"][(curr_state, event_type)]`
   is accessed. A miss produces `ErrorMicroStep(error_kind="cant_happen")`. An
   entry with `next_state=None` and no functions is treated as event-ignored and
   processing stops silently.

7. **Guard evaluation.** If `guard_fn` is present it is called with `(instance,
   args)`. A `GuardEvaluated` micro-step is yielded with the boolean result. If the
   guard returns `False`, processing stops — no state change, no action, no
   `TransitionFired`.

8. **Run-to-completion lock.** `self._in_flight = event` is set immediately before
   calling `action_fn`. Any `generate()` calls made by the action will see
   `_in_flight is not None` and route into `_pending_generated`.

9. **Action execution.** `action_fn(instance, args, scheduler)` is called. A
   two-argument fallback (`action_fn(instance, args)`) is tried if the three-arg
   call raises `TypeError`, supporting simpler action signatures. The return value,
   if a `dict`, is captured as `assignments_made` in the `ActionExecuted` micro-step.

10. **State transition.** `registry.set_state()` updates `curr_state`. A
    `TransitionFired` micro-step is yielded.

11. **Final state check.** If `next_state` is in `cdef["final_states"]`,
    `registry.delete_async()` is called. The resulting `__deletion__` event is
    appended directly to `_pending_generated` (not re-entered through `enqueue()`
    to avoid double-buffering).

12. **Buffer flush.** The `finally` block runs unconditionally:
    - `_pending_generated` is swapped out and cleared.
    - `_in_flight` is set to `None`.
    - Each buffered event is passed to `_enqueue_direct()`, which routes it to
      `_priority` or `_standard` without re-checking `_in_flight` (which is now
      `None`).

    The flush happens even if the action raises an exception, preventing
    `_in_flight` from remaining set and permanently blocking future generates.

## 3. Polymorphic Dispatch

`_resolve_target(target_class, target_id)` implements D-20.

```
_resolve_target(target_class, target_id):
    if registry.lookup(target_class, target_id) is not None:
        return (target_class, target_id)   # direct hit
    for sub in generalizations.get(target_class, []):
        if registry.lookup(sub, target_id) is not None:
            return (sub, target_id)        # subtype hit
    return (None, None)                    # no match
```

`generalizations` is a `dict[str, list[str]]` passed into the scheduler at
construction time (from `DomainManifest.generalizations`). The keys are supertype
names; the values are ordered lists of known subtypes.

When the target is found via a subtype, `_process_event()` uses the subtype's
`ClassManifest` for all subsequent table lookups. The event's original
`target_class` is irrelevant after this point.

**No-match behavior:** `_resolve_target` returning `(None, None)` causes an
`ErrorMicroStep(error_kind="unknown_target")`. The event is silently dropped after
the error step is yielded.

**Supertype instances never exist.** `InstanceRegistry.create_sync/async` rejects
instantiation of abstract classes (`is_abstract=True`) with
`ErrorMicroStep(error_kind="abstract_instantiation")`. `generalizations` entries
are therefore always checked against concrete subtype registries.

## 4. Implicit Transitions

An implicit transition is a transition table entry where `next_state` resolves to
a state different from `curr_state` but no explicit action is defined —
`action_fn` is `None`.

The scheduler handles this transparently: `action_fn` being `None` simply skips
the action execution block. `_in_flight` is still set (and cleared in `finally`),
the state is still advanced, and `TransitionFired` is still yielded. No special
branch handles the nil-action case.

**Event-ignored entries** are distinct: an entry with `next_state=None`,
`action_fn=None`, and `guard_fn=None` is the explicit "ignore this event in this
state" marker. The scheduler returns early without yielding any transition or
action steps.

**Guard on an implicit transition:** If `guard_fn` is present and returns `False`,
neither the state change nor the action executes — the same as for any guarded
transition. The guard is evaluated before `_in_flight` is set, so a guard
rejection never touches the run-to-completion buffer.

## 5. Async Lifecycle Events

### Instance creation

`ctx.create_async()` calls `registry.create_async()`, which:
1. Allocates the instance dict in the registry immediately with `curr_state=None`.
2. Returns an `InstanceCreated(mode="async")` micro-step.
3. Returns an `Event(event_type="__creation__", ...)` carrying `initial_state` in
   `args`.

`ctx.create_async()` then calls `scheduler.enqueue(evt)`. If called from within an
action (i.e., `_in_flight is not None`), the `__creation__` event is buffered in
`_pending_generated`. The instance exists in the registry from the moment
`create_async` returns, but its state is `None` and no entry action has run. The
`__creation__` event's processing (step 3 in section 2) sets `curr_state` and runs
any `(None, "__creation__")` entry action.

### Instance deletion

`ctx.delete(sync=False)` calls `registry.delete_async()`, which:
1. Checks the instance exists.
2. Returns an `Event(event_type="__deletion__", ...)` without modifying the
   registry.

The event is then passed to `scheduler.enqueue()`. If called during an action,
it enters `_pending_generated`. The instance remains live in the registry until
the `__deletion__` event is processed, at which point `registry.process_deletion()`
removes it and yields `InstanceDeleted(mode="async")`.

This means action code executing after an async delete call can still read the
instance's attributes — the instance is not gone yet.

### Final-state auto-deletion

When a transition lands on a final state, the scheduler calls
`registry.delete_async()` and appends the resulting event directly to
`_pending_generated`. This bypasses `enqueue()` to avoid a second round of
`_in_flight` checks. The deletion event is flushed alongside any other buffered
events at the end of the `finally` block.

## 6. Clock and Delay Queue Interaction

`SimulationClock` tracks simulation time in milliseconds as a plain float. It has
no dependency on wall-clock time; the caller is responsible for calling
`clock.advance(delta_ms)` to move time forward.

### Expiry calculation

When `_enqueue_delay()` runs:

```python
expiry = self._clock.now() + (event.delay_ms or 0.0)
```

`clock.now()` returns the current simulated time. The expiry is an absolute
simulation timestamp, not a countdown.

`tick_delay_queue()` promotes events where `expiry_ms <= clock.now()`. Because
`clock.advance()` multiplies `delta_ms` by `speed_multiplier` before adding:

```python
self._time_ms += delta_ms * self._speed_multiplier
```

a `speed_multiplier > 1.0` causes the clock to advance faster than real-time,
making delay expirations occur sooner in real elapsed time. A multiplier of `0.5`
slows the clock. The multiplier has no effect on action execution speed — it only
governs when `clock.now()` crosses an expiry threshold.

### Pause and resume

`clock.pause()` sets `_paused = True`. While paused, `clock.advance()` is a no-op:
the internal `_time_ms` does not change. `tick_delay_queue()` is still called at
the top of the execution loop, but because `clock.now()` returns a stale value, no
delay events will expire. Events already promoted to the standard queue before the
pause are unaffected — they will be dispatched normally.

`clock.resume()` clears `_paused`. The next `advance()` call resumes time
accumulation from wherever `_time_ms` was frozen.

There is no "catch-up" on resume: wall-clock time that elapsed during a pause is
not injected. The GUI debugger (planned for phase 5.4/5.5) uses pause/resume to
implement step-through mode without perturbing delay timing.

### At-most-one invariant (D-14)

Before inserting a delayed event, `_enqueue_delay()` scans the heap for an
existing entry with the same `(event_type, sender_class, sender_id, target_class,
target_id)` triple. If found, that entry is removed before the new one is pushed.
This enforces the rule that at most one delayed signal of a given type per
sender-receiver pair may be pending at any time.

## 7. Error Handling

Per D-28, runtime errors are yielded as `ErrorMicroStep` records rather than raised
as exceptions. This keeps the micro-step stream intact for observers (CLI, GUI) and
allows simulation to continue past local errors without crashing the generator.

### `ErrorMicroStep` fields

```python
@dataclass(frozen=True)
class ErrorMicroStep(MicroStep):
    error_kind: str    # discriminator
    message: str       # human-readable description
    context: dict      # structured payload for the caller
    type: Literal["error"]
```

### Error kinds

| `error_kind` | When emitted |
|---|---|
| `unknown_target` | Target instance not found, and no matching subtype instance exists |
| `unknown_class` | `target_class` not in `class_defs` |
| `cant_happen` | `(curr_state, event_type)` not in the transition table |
| `guard_error` | `guard_fn` raised an exception |
| `abstract_instantiation` | Attempt to instantiate a class with `is_abstract=True` |
| `double_deletion` | `delete_sync` or `process_deletion` called on a non-existent instance |

### Why errors don't raise

The generator pattern (`yield from`) means any unhandled exception propagates
immediately to the consumer and terminates the stream. An `ErrorMicroStep` instead
allows the consumer to decide: log and continue, break on first error, or collect
all errors and report at the end. The CLI test harness and GUI debugger need this
flexibility.

Guard errors specifically catch `Exception` around the guard call to prevent
poorly-written guard code from aborting an otherwise valid simulation. The guard is
treated as returning `False` on error, and the error step is emitted before
returning.

## Appendix: Key Data Structures

**`ClassManifest`** (from `engine/manifest.py`) — the runtime-compiled view of a
single class. The scheduler only reads three fields: `transition_table`,
`final_states`, and `initial_state`. The registry additionally reads `attributes`,
`identifier_attrs`, `supertype`, and `is_abstract`.

**`transition_table: dict[tuple[str | None, str], TransitionEntry]`** — keyed by
`(curr_state, event_type)`. The `None` key for `curr_state` is used exclusively by
`__creation__` events to store the optional entry action.

**`DomainManifest.generalizations: dict[str, list[str]]`** — maps each supertype
name to its ordered list of concrete subtypes. Built by the compilation pipeline;
consumed directly by the scheduler constructor.

**`make_instance_key(identifier: dict) -> frozenset`** — converts an identifier
dict to an order-independent frozenset for use as a dict key. All registry and
scheduler code uses this function to normalize identifier comparisons.
